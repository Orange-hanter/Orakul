# Сценарий демо для пилотных клиентов — R2 Supplier MVP

## Назначение

Документ — рабочая инструкция для проведения демо новой функциональности (US-09 каталог, US-10 поиск аналогов, US-11 мастер заявок) перед пилотным клиентом или потенциальным покупателем. Цель встречи — закрыть **Gate G2: Pilot → Paid конверсию**.

**Длительность:** 15–20 минут на демо + 10–15 минут на Q&A.
**Формат:** screen share (Zoom/Google Meet) или iPad в офисе клиента.
**Кто присутствует:** Tech Lead (демо) + Product Owner (комментарии по ценности). Со стороны клиента: собственник + закупщик/управляющий.

---

## Подготовка (за 24 часа)

### 1. Развёртывание чистого инстанса

Лучше всего показывать на отдельном инстансе (не на проде клиента, чтобы данные не пересекались).

```bash
# На рабочей машине либо staging-сервере
cd /opt/orakul         # или путь к рабочему чекауту
git pull
npm run build
systemctl restart orakul.service   # или npm start
```

### 2. Загрузка демо-данных

```bash
# Из директории app/
APP_PASSWORD=<пароль_инстанса> npm run seed-demo

# Либо против удалённого инстанса:
APP_PASSWORD=<пароль> npm run seed-demo -- --url http://demo.orakul.app
```

После выполнения скрипта в инстансе появятся:
- 12 товаров склада (мука, масло, мясо, овощи, кофе, и т.д.)
- 3 поставщика: ООО АгроПоставка, ИП Козлов, ООО ФрешМаркет
- 14 позиций каталога с пересечениями (мука и сахар есть и у АгроПоставки, и у Козлова)
- История цен на муку, масло, кофе (3 точки на каждую — видимый тренд)
- 2 заявки: черновик у АгроПоставки + отправленная у Козлова

### 3. Проверка чек-листа за 5 минут до демо

- [ ] iPad / браузер залогинены, токен свежий
- [ ] Открыт URL приложения, не дев-сервер с консолью
- [ ] Уведомления и Telegram-чаты выключены
- [ ] Закрыты все лишние вкладки
- [ ] Зум на 110%, чтобы цифры читались на видеозвонке
- [ ] Открыт slide с Business Case (резервный для блока «4. Цифры»)

## Сценарий встречи

### Открытие (2 мин)

> **Скажите:** «Сегодня покажем три новых модуля, которые мы вытащили в Q2 из плана Q3 — потому что именно они дают самый крупный финансовый эффект для вашей сети. Это каталог поставщиков, поиск более выгодных предложений и пошаговое оформление заявок. Сразу после демо обсудим, как мы это включаем в ваш пилот.»

**Цель открытия:** установить контекст «вы видите то, что раньше было в Q3, и теперь это доступно сейчас». Это контр-ход против возражения «давайте подождём релиза».

---

### Блок 1 — Боль (1 мин, без экрана)

Не показывайте экран. Спросите:

> «Расскажите коротко — как сегодня выглядит процесс закупки у вас? Кто решает, у кого покупать? Где хранятся прайс-листы? Когда последний раз вы сравнивали цены трёх поставщиков на одну и ту же позицию?»

**Что ожидаем услышать (типовые ответы):**
- «Прайс-листы в почте / WhatsApp / голове закупщика»
- «Сравниваем редко, когда вырастают цены»
- «Заказ оформляем в чате или по телефону»

**Когда получили ответ, перейдите:**

> «То, что мы сейчас покажем, решает все три эти проблемы. Открываем приложение.»

---

### Блок 2 — Каталог поставщиков (4 мин)

#### Шаг 2.1. Список поставщиков

1. Откройте вкладку **«Поставщики 🏪»**.
2. Скажите:
   > «Это все ваши поставщики в одном месте. У каждого видны категории и количество позиций. Поиск работает по названию или по категории — попробуйте ввести "молочка".»
3. Введите в поиск: `молочка`. Покажите, что выпали только релевантные поставщики.
4. Очистите поиск.

#### Шаг 2.2. Карточка поставщика и тренд цены

1. Нажмите на **ООО АгроПоставка**.
2. Скажите:
   > «Здесь видно весь ассортимент поставщика — позиция, текущая цена, единица. А вот это (показать бейдж) — тренд за последние изменения. ▲ красный значит цена выросла, ▼ зелёный — упала.»
3. Покажите карточку **«Мука в/с 25 кг»** (видна красная ▲ +4.3%).
4. Скажите:
   > «И посмотрите вот сюда — система автоматически нашла, что у другого поставщика та же мука дешевле на 12.5%. Это та самая 💡 подсказка.»

> 💡 **Триггер «вау»:** именно здесь клиент чаще всего говорит «о, а это работает?» — после этого всё проще.

#### Шаг 2.3. История цен и аналоги

1. Нажмите на карточку **«Мука в/с 25 кг»**.
2. Откроется детальная модалка.
3. Скажите:
   > «История цен — вся, с датами и процентом изменения. Любая цена меняется — здесь автоматически появляется запись. Никто ничего не забудет.»
4. Прокрутите вниз до раздела «Аналоги».
5. Скажите:
   > «А здесь — все другие поставщики, у кого есть та же позиция. Отсортировано по цене. ИП Козлов даёт 1.05 BYN/кг против ваших 1.20 — это минус 12.5%. На годовом объёме муки у вас это будет (показать на бумаге или slide) около 1500–3000 BYN экономии — только на одной позиции.»
6. Закройте модалку.

---

### Блок 3 — Мастер заявок (5 мин)

#### Шаг 3.1. Список заявок

1. Откройте вкладку **«Заявки 📝»**.
2. Скажите:
   > «Все заявки — здесь. Активные сверху, фильтр сверху. Видите эту (показать) — она в статусе "Отправлена" Козлову, заметка: "Переключились с АгроПоставки — экономия 14% на бакалее". Это пример того, как закупщик использует подсказку из каталога.»

#### Шаг 3.2. Создание новой заявки (главный wow-блок)

1. Нажмите **«+»** в правом нижнем углу.
2. **Шаг 1 — Поставщик.** Скажите:
   > «Выбираем поставщика. Приостановленных не видно, фильтр срабатывает автоматически — нельзя ошибиться.»
   Выберите **ИП Козлов**.
3. **Шаг 2 — Позиции.** Скажите:
   > «Видны все позиции этого поставщика. Указываем сколько нужно. Вот эта (показать минимальную партию) — система подскажет, если введёте меньше минимума.»
   Введите:
   - Мука пшеничная высш. сорт → **50**
   - Сахар-песок ГОСТ → **30**
   Покажите, что внизу появился счётчик «2 позиции / 117.00 BYN».
4. **Шаг 3 — Обзор.** Скажите:
   > «Полный обзор заказа. Выбираем дату доставки, добавляем заметку. Жмём «Создать заявку».»
   Дата доставки — на 2 дня вперёд. Заметка: `Демо-заказ`.
5. Нажмите **«Создать заявку»**.
6. Покажите, что в списке появилась новая заявка с номером **ОРД-2026-XXXX**:
   > «Номер сгенерирован автоматически. Заявка в черновике. Когда закупщик отправит её — нажимает "Отправить" в карточке. Когда поставщик привёз — "Принято".»

#### Шаг 3.3. Жизненный цикл заявки

1. Откройте только что созданную заявку.
2. Нажмите **«📤 Отметить отправленной»** → бейдж сменился.
3. Покажите кнопку **«✅ Поставка принята»** (не нажимайте — оставьте на «отправлена», чтобы клиент видел рабочий процесс).
4. Закройте.

---

### Блок 4 — Цифры (3 мин)

Откройте slide с Business Case ([docs/06-financial/01-business-case-and-roi.md](../06-financial/01-business-case-and-roi.md) §2) или нарисуйте на бумаге:

```
Точка с оборотом 5 млн ₽/мес:
  Food cost (35%)        = 1.75 млн ₽/мес
  Экономия на закупках (10%) = 175 тыс ₽/мес
  За год                  = ~2.1 млн ₽/год
```

> **Скажите:** «Это не теория. Каждая позиция, где аналог дешевле — это сейчас цифра в вашем приложении. Не нужно ждать квартал, чтобы понять, есть ли эффект — он виден на каждой карточке поставщика.»

**Готовый ответ на «а вдруг качество другое»:**
> «Поэтому в системе у каждого поставщика рейтинг — он копится автоматически по принятым заявкам. И каждое переключение остаётся в истории — никаких "тёмных" решений.»

---

### Блок 5 — Что в следующем релизе (2 мин)

Открыть slide с дорожной картой или [Roadmap §4](01-roadmap.md):

> «То, что вы видели — это MVP. В августе добавим: импорт прайсов из xlsx одним кликом, рейтинги поставщиков, fuzzy-поиск аналогов (когда одна и та же мука называется у разных поставщиков по-разному). В сентябре — финансовая аналитика с P&L и сравнение точек по сети.»

---

### Закрытие (2 мин)

Три варианта закрытия по уровню сигнала со стороны клиента:

**Если клиент в восторге** («когда можем включить?»):
> «Можем включить эту функциональность в ваш текущий пилот в течение 3 рабочих дней — у нас уже есть готовый сетап. Дальше две недели вы пользуетесь, мы делаем замер экономии по факту, и обсуждаем переход на платную подписку. Сегодня договоримся по дате включения?»

**Если клиент молчит/думает** («интересно, мы подумаем»):
> «Хорошо. У меня будет один вопрос: какой кейс/цифру вам нужно увидеть, чтобы принять решение? Если "экономия 100 тыс ₽ за 2 недели на вашем объёме" — это конкретная задача, мы её можем поставить как цель пилота, замерим и покажем результат через две недели.»

**Если клиент против** («дорого / не сейчас / у нас другая система»):
> «Понимаю. Тогда последний вопрос: что мешает больше — цена, риск перехода или загруженность команды? Этот ответ поможет нам в будущем сделать решение под ваш случай.» (Это для квалификации, не для продажи в моменте.)

## Анти-паттерны (что НЕ делать)

| Что | Почему |
|-----|--------|
| Открывать вкладки «Стопы», «Меню», «Стоп-листы» во время демо новой функциональности | Размывает фокус. Эти вкладки — для существующего пилота, новые клиенты их видят отдельно. |
| Показывать чистый инстанс без seed-данных | Пустые экраны → клиент не понимает, что увидит у себя. |
| Зачитывать тексты со спецификации | Спецификация для разработчиков, не для клиента. На демо — только функциональные демонстрации. |
| Обсуждать технологии (React, Node, шифрование) без явного запроса | Снижает фокус на ценности. Если спросят — кратко: «Стек проверен на пилотах, безопасность через AES-256 шифрование данных». |
| Открывать DevTools / консоль | Заставляет клиента сомневаться: «зачем им это нужно?» |
| Обещать сроки на лету («сделаем за неделю») | Все обещания только из подписанного [Roadmap](01-roadmap.md). |

## Чек-лист пост-демо (в течение 2 часов после)

- [ ] Записать в [Decision Log](../05-governance/05-decision-log.md) ключевые реакции клиента (что зашло, что нет)
- [ ] Если клиент дал зелёный свет → создать запись о включении модуля в его пилот с датой
- [ ] Если клиент дал возражение → зафиксировать в [Risk Register](../05-governance/01-risk-register.md) или в карточке клиента в CRM
- [ ] Сделать follow-up письмо с ссылкой на ваш URL + краткое summary показанного

## Метрики качества демо (для самооценки)

| Метрика | Норма | Алёрт |
|---------|-------|-------|
| Длительность блока «Каталог» | 3–5 мин | > 7 мин (слишком углубились) |
| Реакция «о, а это уже работает?» в Блоке 2 | да | нет (что-то не сработало визуально) |
| Конкретный вопрос клиента про цены/экономию | да | нет (клиент не подключился) |
| Договорённость по следующему шагу | да | нет (повторное демо не нужно) |
