# Чек-лист предпроектного аудита данных Kiepper

## 0. Зачем

Без качественных данных в Kiepper Orakul не работает — это закон, а не лозунг. Этот чек-лист — **обязательный gating-критерий перехода с этапа E1 на E2**. Без подписанного отчёта по аудиту мы не запускаем рекомендации.

Цели аудита:
1. **Оценить готовность данных** для прогноза, факт vs норма и расчёта маржи.
2. **Выявить критические пробелы** до того, как они проявятся как «галлюцинации» ИИ.
3. **Зафиксировать baseline** для последующего расчёта ROI.
4. **Согласовать план чистки** данных, если это необходимо.

---

## 1. Объём аудита

| Сущность Kiepper | Глубина | Зачем |
|------------------|---------|-------|
| Справочник номенклатуры (SKU) | 100% активных | Базовый каталог |
| Справочник единиц измерения | 100% | Без единого UoM нет рецептур |
| Справочник складов | 100% | Привязка остатков |
| Рецептуры (нормы закладки) | 100% активных блюд за последние 12 мес | Расчёт факт vs норма |
| Историч. продажи (чеки, блюда) | 12 мес минимум, 24 мес желательно | Прогноз спроса |
| История остатков | 12 мес | Backtest прогнозов |
| Списания | 12 мес | Аномалии, baseline |
| Закупки (заказы, поступления) | 12 мес | Оборачиваемость, поставщики |
| Цены (закупочные и продажи) | Все на дату | Маржинальность |
| Структура точек, время работы, часовые пояса | 100% | Уведомления, бизнес-правила |

## 2. Чек-лист критериев качества (с порогами)

### Уровень 1: Блокирующий — без этого нельзя стартовать (BLOCKER)

| # | Что проверяем | Порог приемлемости | Если не проходит |
|---|----------------|---------------------|-------------------|
| 1.1 | Справочник SKU без дублей | ≤2% дублей | Чистка SKU до старта |
| 1.2 | Все активные блюда имеют рецептуру | ≥95% | Sprint чистки 2 нед, либо запуск с урезанным scope (только блюда с рецептурой) |
| 1.3 | Все рецептуры используют справочные единицы измерения | 100% | Конвертация UoM до старта |
| 1.4 | История продаж глубиной | ≥6 мес (минимум для сезонности) | Если <6 мес — невозможен прогноз; продукт не запускается, обсуждается отсрочка или альтернатива |
| 1.5 | Точки в Kiepper связаны со складами | 100% активных | Восстановление связей |
| 1.6 | Часовые пояса точек определены | 100% | Без этого ломаются уведомления |

> **Если хотя бы один блокер не пройден — этап E2 не стартует.** Это часть соглашения с клиентом и фиксируется в kick-off.

### Уровень 2: Жёлтый — можно стартовать с ограничениями

| # | Что проверяем | Порог приемлемости | Что меняется при провале |
|---|----------------|---------------------|---------------------------|
| 2.1 | Цены закупочные актуальны (изменены не позднее 60 дней) | ≥80% SKU | Маржа считается с поправкой; в обоснованиях указывается «цены могут быть устаревшими» |
| 2.2 | Списания привязаны к причине | ≥70% | Аномалии работают, но менее точные |
| 2.3 | Поступления имеют поставщика | ≥90% | US-04 (сравнение) с урезанным scope |
| 2.4 | Заказы поставщикам имеют статусы (черновик/подтверждён/получен) | ≥80% | План vs факт считается с погрешностью |
| 2.5 | Корректировки рецептур имеют дату изменения | 100% за последние 12 мес | Без этого backtest неточен |
| 2.6 | Чеки имеют детализацию по блюдам (не только итог) | ≥95% | Без детализации не работает прогноз спроса |

### Уровень 3: Зелёный — приятно иметь, не блокирует

| # | Что проверяем | Порог |
|---|----------------|-------|
| 3.1 | Категории меню заполнены | ≥90% |
| 3.2 | Аллергены заведены | По возможности |
| 3.3 | Упаковка/штрихкоды у поставщиков | По возможности |
| 3.4 | Себестоимость закладки рассчитывается в Kiepper | По возможности |

## 3. Технические проверки целостности

| # | Проверка | Метод |
|---|----------|-------|
| T-01 | Связность таблиц (FK без orphans) | SQL на тестовом срезе |
| T-02 | Свежесть данных через API (lag) | Запрос timestamp последних 100 записей |
| T-03 | Стабильность схемы (нет недокументированных кастомных полей) | Сверка со стандартной схемой Kiepper |
| T-04 | Производительность выгрузки (полный snapshot за разумное время) | Бенчмарк выгрузки |
| T-05 | Доступы и квоты API подтверждены IT-владельцем Kiepper | Письменное подтверждение |

---

## 4. Бизнес-проверки реалистичности данных

> Эти проверки чаще выявляют проблемы, чем технические. Они отвечают на вопрос «выглядят ли данные как настоящая работа сети?»

| # | Проверка | Сигнал проблемы |
|---|----------|-----------------|
| B-01 | Распределение продаж по часам не имеет «провалов в середине дня» | Возможна потеря чеков в пиковые часы |
| B-02 | Количество ингредиентов на блюдо в среднем 5–15 | <3 — рецептуры неполные; >25 — возможны технические дубли |
| B-03 | Доля списаний к обороту в пределах 1–8% | <0.5% — недозаписывают; >12% — аномалия или ошибка учёта |
| B-04 | Динамика маржинальности без скачков ±20% между неделями | Скачок = смена цен или ошибка |
| B-05 | Совпадение продаж с инвентаризациями ≥90% | Большой разрыв = «теневые» движения |
| B-06 | Наличие пустых смен (нулевые продажи в рабочий день) | Точка не сдала отчёт; пробел в данных |
| B-07 | Среднее количество поставщиков на SKU | =1 для всех — ограниченная конкуренция; >5 — справочник «замусорен» |

## 5. Baseline-замер (обязательная часть аудита)

Перед запуском рекомендаций фиксируется **baseline** на 14 рабочих дней:
- BIZ-01 % списаний к обороту — текущее значение по точке;
- BIZ-02 отклонение факт vs план закупок — текущее;
- BIZ-03 средняя маржинальность меню — текущее;
- BIZ-04 часы на отчётность — измеряется опросом или таймингом;
- BIZ-06 кол-во стоп-листов / нед — журнал управляющего.

> Baseline подписывается финконтролёром клиента. Без подписи дальнейшее измерение ROI не имеет силы (любой может оспорить «мы и без вас были на этих цифрах»).

---

## 6. Артефакты аудита (что выдаём клиенту)

1. **Audit Report (PDF)** — сводка по всем уровням, с цветовой индикацией.
2. **Список dirty data** (CSV) — конкретные SKU/рецептуры/точки, требующие правок.
3. **План чистки данных** — задачи, ответственные клиента, сроки.
4. **Baseline-отчёт** — подписан финконтролёром.
5. **Соглашение об ограничениях** — если стартуем с жёлтой зоной, какие функции временно ограничены.
6. **Go/No-Go запись** — решение управляющего комитета о переходе к E2.
