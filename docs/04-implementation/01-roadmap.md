# Implementation Roadmap

## 1. Принципы roadmap

1. **Roadmap не равен бэклогу.** Здесь — крупные результаты, выраженные в **изменении поведения пользователя или бизнеса**. Технические задачи скрыты внутри.
2. **Каждый этап имеет gating-критерий.** Без выполнения критерия следующий этап не стартует, даже если есть давление по срокам.
3. **Срезы ежемесячно.** На 1-й рабочий день месяца PMO готовит срез: что в плане, что в задержке, какие риски.
4. **Изменения — через CR.** Любое изменение объёма этапа = [Change Request](../05-governance/04-change-request-template.md), решение = [Decision Log](../05-governance/05-decision-log.md).

## 3. Этапы внедрения (бизнес-уровень) — для одного клиента

> Это шаблон, по которому каждый клиент проходит свой жизненный цикл. Соответствует [BRD §8](../02-business/01-business-requirements-document.md).

| Этап | Срок | Gating-критерий перехода к следующему | Артефакты на выходе |
|------|------|----------------------------------------|----------------------|
| **E1. Диагностика и базовые KPI** | 2–3 нед | Карта потерь подписана; baseline зафиксирован 14 дней | Аудит данных Kiepper, baseline-отчёт, согласованные KPI |
| **E2. Пилот на 1–2 точках** | 4–6 нед | ARAR ≥60%; снижение списаний ≥10% от baseline | Калибровка прогнозов, кейс-отчёт, обратная связь |
| **E3. Масштабирование на сети** | 3–4 нед | Все плановые точки подключены, регламенты приняты | Регламенты работы с ИИ, обучение завершено |
| **E4. Стабилизация и передача** | 2–3 нед | KPI стабильно в коридоре 2 месяца; инцидентов P1 нет | Финальный аудит, акт приёмки |
| **E5. Сопровождение** | Постоянно | Ежеквартальный QBR, NRR ≥110% | QBR-отчёты, план развития |

## 4. Дорожная карта продукта (релизы)

### Релиз R1 — MVP Foundation
**Срок:** конец Q1 2026  
**Цель релиза:** обеспечить пилот у двух клиентов с минимально достаточным функционалом и измеримой ценностью.

| Включено | Связь с US |
|----------|-----------|
| Алёрты по остаткам и срокам годности | US-01 |
| Сравнение факт vs норма по сменам | US-02 |
| Рекомендации заказа (черновик в Kiepper) | US-03 |
| Маржинальность блюда в реальном времени | US-05 |
| Сетевой дашборд (минимальный) | US-07 |
| Объяснимость рекомендации (обоснование + 👍/👎) | US-08 |
| Интеграция Kiepper: справочник, остатки, продажи, рецептуры, закупки | — |
| Аутентификация и роли (RBAC) | — |
| Журнал событий (audit-trail) | — |

**Gate G1:**
- ARAR ≥60% на двух пилотных точках.
- BIZ-05 (MAPE) ≤15% на топ-50 ингредиентов.
- 100% рекомендаций имеют обоснование.

### Релиз R2 — Stabilization & Health + Supplier MVP
**Срок:** середина Q2 2026
**Цель:** Pilot → Paid конверсия ≥80%; раскрытие Value Driver V2 (оптимизация закупок).

| Включено | Связь |
|----------|-------|
| Health Score клиента | KPI HLT-01..04 |
| Модуль обратной связи и корректировок | US-08 |
| Улучшение UX управляющего на мобильном | Persona Manager |
| Документация обоснований модели | Принцип Always Explain |
| **Каталог поставщиков (MVP — ручной ввод цен)** | **US-09 (P0, перенесён из R3 — [CR-2026-001](../05-governance/cr-2026-001-supplier-catalog-to-r2.md))** |
| **Поиск аналогов (точное совпадение по справочнику)** | **US-10 (P0, перенесён из R3)** |
| **Мастер заявок поставщикам (упрощённый)** | **US-11 (P0, перенесён из R3)** |

**Gate G2:**
- Pilot → Paid ≥80%.
- Снижение списаний ≥15% от baseline на двух пилотах.
- NPS управляющего ≥+30.
- **Каталог поставщиков работает на 2 пилотах с минимум 3 поставщиками × 20 позиций.**

### Релиз R3 — Network Scale
**Срок:** середина Q3 2026
**Цель:** удвоение точек у существующих клиентов и подключение новых сетей.

| Включено | Связь |
|----------|-------|
| Расширение каталога: импорт xlsx, рейтинги, fuzzy-поиск аналогов | US-09/US-10 (P1, продолжение R2 MVP) |
| Сценарное моделирование меню (US-06) | P1, тариф Growth+ |
| Onboarding playbook (тиражируемая методика) | OPS-03 (TTV) |
| Снижение TTV до ≤21 дня | OPS-03 |

**Gate G3:**
- ARAR ≥80% в среднем по платящим.
- TTV новых клиентов ≤21 дня.
- NRR ≥110% по квартальной когорте.

### Релиз R4 — Year-End Strong
**Срок:** конец Q4 2026  
**Цель:** закрыть годовой план; подготовиться к расширению на столовые/корп. питание в 2027.

| Включено | Связь |
|----------|-------|
| Адаптация под тендеры (закупки больших объёмов) | Сегмент столовых |
| Аномалии 2-го уровня (повторяющиеся паттерны) | US-02 расширение |
| SLA Uptime ≥99.5% (формализация) | OPS-01 |
| Готовность ко второму типу клиента | GTM Sectoral |

**Gate G4:**
- 15 логотипов / 50 точек.
- Logo retention 12-month ≥90%.
- 3 предконтракта на Q1 2027.

## 5. Зависимости между этапами

| Что зависит от чего | Влияние |
|---------------------|---------|
| Релиз R2 от партнёрства с Kiepper (для канала продаж) | Если Kiepper-меморандум не подписан к концу Q1, переходим на plan B (усиление прямых продаж) — задержка GTM на 4–6 нед |
| Релиз R3 от стабильности интеграции с Kiepper | Любые breaking changes API Kiepper = блокер до митигации |
| Релиз R4 от наличия минимум 3 публичных кейсов | Сегмент столовых требует социального доказательства |
| Все релизы от качества данных в Kiepper у клиента | Без чистых рецептур и баз — никакой ИИ не помогает |

## 7. Риски roadmap (топ-5; полный — в Risk Register)

| Риск | Влияние на roadmap | Митигация |
|------|---------------------|-----------|
| Качество данных Kiepper у клиентов | Сдвиг G1 на 2–4 нед | Жёсткий Data Audit перед стартом |
| Сопротивление персонала (низкий ARAR) | Не пройдём G1/G2 | Принципы объяснимости + Change Mgmt + чемпионы |
| Партнёрство Kiepper не подписано в Q1 | Замедление GTM, задержка G2 | Plan B: усиление прямых продаж + Kiepper-независимый коннектор |
| Kiepper выпускает свой ИИ-модуль | Требует переоценки позиционирования | Market Watch + договорная защита партнёрством |
| Нехватка ресурса CSM при росте | Падает retention, не пройдём G3 | План найма CSM + playbook автоматизации |

> Полный список — [Risk Register](../05-governance/01-risk-register.md).
