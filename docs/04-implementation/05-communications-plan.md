# Communications Plan

## 0. Принципы

1. **Минимум каналов, максимум регулярности.** Чем больше каналов — тем выше шанс, что важное сообщение потеряется.
2. **Дайджест важнее push'ей.** Короткий регулярный дайджест работает лучше потока уведомлений.
3. **Один источник правды.** Решения и статусы — в этой документации; чаты — для оперативки, не для решений.
4. **Тишина — тоже информация.** Если регулярная встреча не нужна — отменяем явно, а не «не пришли».
5. **Цикл «событие → коммуникация → решение → закрытие».** Каждое сообщение — либо запрос действия, либо отчёт о результате.

## 1. Каналы и их назначение

| Канал | Для чего | Не для чего |
|-------|----------|-------------|
| **Email** | Формальные решения, контракты, отчёты, утверждения | Оперативные вопросы, чаты |
| **Внутренний мессенджер (Slack/Telegram)** | Оперативный канал команды, обсуждение задач | Принятие финальных решений (фиксируем в документ) |
| **Календарные встречи (Zoom/очно)** | Решения, требующие обсуждения; QBR; ритуалы | Информирование (вместо встречи — рассылка) |
| **Документация (этот репозиторий)** | Решения, требования, артефакты | Оперативные вопросы |
| **CRM/CSM-инструмент** | История работы с клиентом | Решения по продукту |
| **In-app уведомления Orakul** | Алёрты пользователям | Маркетинг, опросы |

## 2. Cadence — регулярные встречи и рассылки

### 2.1 Внутренние

| Активность | Частота | Длит. | Кто | Цель | Артефакт |
|-----------|---------|-------|-----|------|----------|
| Daily standup команды | Ежедневно | 15 мин | PO, DL, TL, DES, CSM | Синхронизация по задачам, блокеры | Заметки в чате |
| PO + DL + TL еженедельно | Еженедельно | 30 мин | PO, DL, TL | Roadmap, релизы, инциденты | Update в roadmap |
| Sales review | Еженедельно | 45 мин | SAL, MKT, CEO | Воронка, сделки, CAC | Сводка по pipeline |
| CSM review | Еженедельно | 45 мин | CSM, PO, PMO | Здоровье клиентов, эскалации | Health Score сводка |
| Weekly Business Review (WBR) | Еженедельно | 60 мин | Спонсор, PO, CSM, SAL | Все KPI, отклонения | WBR-отчёт |
| Monthly Business Review (MBR) | Ежемесячно | 90 мин | Управляющий комитет | Полный обзор KPI, риски | MBR-протокол + Decision Log |
| Quarterly retrospective + OKR-планирование | Ежеквартально | 4 ч | Все команды | Что сработало, OKR на след. квартал | Новые OKRs |

### 2.2 Клиентские (на каждом активном клиенте)

| Активность | Частота | Длит. | Кто | Цель |
|-----------|---------|-------|-----|------|
| Kick-off | Однократно (старт) | 90 мин | Все ключевые роли клиента + CSM, PMO | Цели, сроки, ответственные |
| Pilot daily check-in | Ежедневно (только в первые 2 недели пилота) | 15 мин | CSM + управляющий пилотной точки | Любые блокеры, обратная связь |
| Pilot weekly review | Еженедельно (E2) | 30 мин | CSM + опердир + управляющий | Прогресс по KPI, обратная связь |
| Status report (письменный) | Еженедельно | — | CSM → клиент | Сводка KPI, риски, действия |
| MBR с клиентом | Ежемесячно | 60 мин | CSM, PO, опердир, финконтролёр | KPI, эскалации, расширение |
| QBR с клиентом | Ежеквартально | 90 мин | Управляющий комитет с обеих сторон | Стратегия, продление, кейс |

### 2.3 Внешние (партнёры)

| Активность | Частота | С кем |
|-----------|---------|-------|
| Партнёр-встреча с Kiepper | Ежемесячно | E-01 |
| Совместный вебинар (маркетинг) | Ежеквартально | Kiepper |
| Отчёт регулятору / комплаенс | По требованию | Регулятор |

```
- Списания к обороту: 3.1% (цель ≤2.8%)
- ...

✅ ЧТО СДЕЛАНО
- ...

⚠ РИСКИ И БЛОКЕРЫ
- ...

🎯 ПЛАН НА СЛЕД. НЕДЕЛЮ
- ...

📅 СЛЕД. КОНТРОЛЬНАЯ ТОЧКА: [дата]

С уважением, [CSM]
```

### 3.2 Эскалация P1 инцидента

```
Тема: 🔴 P1 ИНЦИДЕНТ — [короткое описание]

ВРЕМЯ ОБНАРУЖЕНИЯ: [HH:MM]
ВЛИЯНИЕ НА КЛИЕНТА: [что не работает / для кого]
ТЕКУЩИЙ СТАТУС: [расследование / митигация / восстановлено]
ОТВЕТСТВЕННЫЙ: [имя]
ETA ВОССТАНОВЛЕНИЯ: [HH:MM или «расследование»]

ПЕРВЫЕ ДЕЙСТВИЯ:
- ...

СЛЕДУЮЩЕЕ ОБНОВЛЕНИЕ ЧЕРЕЗ: 60 минут.
```

### 3.3 Decision-запрос

```
Тема: [DECISION REQUIRED] — [решение, которое нужно принять]

КОНТЕКСТ
[2–3 предложения]

ОПЦИИ
A) ... (плюсы / минусы)
B) ...
C) ...

РЕКОМЕНДАЦИЯ
[A/B/C] потому что ...

ЧТО НУЖНО
✅ Утвердить опцию до [дата]
❓ Альтернативные предложения принимаются до [дата]

ПОСЛЕ РЕШЕНИЯ
Решение фиксируется в Decision Log [ссылка].
```

## 5. Управление шумом и ритмом

### 5.1 Anti-patterns
| Антипаттерн | Что делаем |
|-------------|------------|
| «Reply-all» цепочки | Запрещены; вынос в чат или встречу |
| Встречи без повестки | Отменяются автоматически |
| Push-уведомления >5 в день на пользователя | Объединяются в дайджест (см. [PRD §9](../03-product/01-product-requirements-document.md)) |
| Решения в чате без фиксации | Сразу выносятся в Decision Log |
| Дублирование (email + чат + встреча) | Один канал на сообщение |

### 5.2 Тихие часы
- Никаких клиентских коммуникаций до 8:00 и после 21:00 локального времени, кроме P1 инцидентов.
- Внутри команды — никаких сообщений в выходные, кроме P1.

## 6. Кризисная коммуникация

### 6.1 Триггеры
- Утечка данных (R-06).
- Падение системы >2ч в часы работы (нарушение SLA).
- Серьёзная ошибка ИИ, повлёкшая убытки клиента.
- Отзыв продукта Kiepper / расторжение партнёрства.
- Регуляторный инцидент.

### 6.2 Регламент
1. **Час 0:** обнаружение → немедленное уведомление CEO + SEC + PMO.
2. **Час 1:** формирование штаба (CEO, PO, TL, SEC, CSM, юрист). Один точечный канал коммуникации.
3. **Час 2:** первое внешнее уведомление пострадавшим (черновик готовит CSM, утверждает CEO + юрист).
4. **Каждые 2 часа** — обновление пострадавшим до закрытия инцидента.
5. **24 часа после закрытия** — post-mortem и публичный отчёт (если применимо).
6. **5 рабочих дней** — отчёт в управляющий комитет с уроками и изменениями процессов.

### 6.3 Кто говорит публично
- С клиентом — CEO + CSM (никто другой).
- С регулятором — юрист.
- Со СМИ — CEO (никто другой).
- Молчание со стороны команды — обязательно.
