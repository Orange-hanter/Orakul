## Описание

Подтвердить, что события списаний доступны через API с указанием причины, количества, ответственного. Критично для модуля выявления потерь и аномалий (BRD §6.5).

## Критерии приёмки

- [ ] Найден API-метод для получения списаний (или зафиксировано отсутствие)
- [ ] Получен образец данных за период (≥7 дней)
- [ ] Описаны доступные поля: ингредиент, количество, дата, причина, ответственный, тип списания (брак / порча / сотрудничество и т.д.)
- [ ] Подтверждена возможность фильтрации по периоду и типу
- [ ] Результат зафиксирован в [Decision Log](../../../05-governance/05-decision-log.md)

## Риски и блокеры

- **Высокий риск:** в публичной документации Quick Resto API списания как отдельный тип события не выделены. Могут быть частью `storage.getIngredientHistory` — нужно проверить структуру и поля ответа.
- Если списания недоступны как явное событие, можно компенсировать дельтой «прогнозный расход vs фактический остаток» — но без указания причины и ответственного. Это снижает ценность модуля аномалий.
- Этот результат — главный вход в TASK-006 (gate-решение).
