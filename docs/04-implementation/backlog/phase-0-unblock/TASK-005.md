## Описание

Измерить технические характеристики Quick Resto API: глубину истории, rate limits, среднюю задержку. Эти параметры определят:
- Частоту polling в [Integration Specification](../phase-1-tech-docs/TASK-012.md)
- Архитектурные решения (нужно ли локальное хранилище для истории)
- Реалистичность SLA из BRD §7 (15-мин latency)

## Критерии приёмки

- [ ] Измерена максимальная глубина истории по каждой сущности:
  - Продажи (transactions)
  - Остатки (storage balances)
  - Движения ингредиентов (ingredient history)
  - Стоп-лист (если есть история)
- [ ] Измерены rate limits: RPS (запросов в секунду), дневной лимит, размер пагинации
- [ ] Измерена средняя задержка ответа для ключевых endpoints (10 запросов × 3 сценария)
- [ ] Подтверждена возможность пакетной выгрузки за период (а не только онлайн-чтения)
- [ ] Результаты зафиксированы в [Decision Log](../../../05-governance/05-decision-log.md)

## Риски и блокеры

- Если глубина истории <1 года — затруднено обучение моделей прогноза (нужны сезонные паттерны).
- Если rate limits жёсткие (<10 RPS) — потребуется агрессивное кэширование и батчевая выгрузка.
- Если задержка >2 сек — невозможно уложиться в SLA «открытие экрана ≤2 сек» (PRD §10).
