# Delivery Backlog — карта модулей

> **Назначение.** Описывает структуру всех модулей бэклога, порядок их выполнения и кросс-зависимости. Это **точка входа** в бэклог: отсюда видно картину целиком, отдельные задачи лежат в папках модулей.

## Кросс-зависимости модулей

```
Phase 0 ──────► Phase 1 ──────► Phase 2 ──────► Phase 6 ──► Sprint 0
   │                              │ ▲
   │                              │ │ (Phase 3-5 идут параллельно
   │                              │ │  и сходятся в Phase 6)
   │                              ▼ │
   │            ┌─── Phase 3 (тестирование) ───┐
   │            │                              │
   │            ├─── Phase 4 (окружения) ──────┤──► Phase 6
   │            │                              │
   └───────────►└─── Phase 5 (пилот) ──────────┘
```

### Жёсткие зависимости

| Зависимая фаза | Зависит от | Почему |
|----------------|------------|--------|
| Phase 1 | Phase 0 (TASK-006) | TRD/SAD пишутся с учётом подтверждённых возможностей API |
| Phase 2 | Phase 1 (TASK-015) | Декомпозиция возможна только когда тех. требования утверждены |
| Phase 3 | Phase 1 (TASK-010, 011, 012) | Test Plans пишутся под TRD и Integration Spec |
| Phase 4 | Phase 1 (TASK-011) | Env Plan базируется на архитектуре из SAD |
| Phase 5 | Phase 0 (TASK-006) | Выбор пилотной точки требует подтверждённого скоупа |
| Phase 6 | Все Phase 1–5 | Финальная проверка готовности |

### Параллельные зависимости (Phase 3, 4, 5 ↔ Phase 2)

Phase 3 (тестирование), Phase 4 (окружения) и Phase 5 (пилот) идут **параллельно** с Phase 2 (декомпозиция). Они зависят от Phase 1, но не от Phase 2 — это позволяет сжать общий срок с 8 до 5 недель.

### Условные зависимости

| Задача | Условие выполнения |
|--------|---------------------|
| TASK-008 (CR к BRD) | Только если TASK-003 или TASK-004 закрылись результатом «API недоступен» |
| TASK-032 (NDA + DPA пилота) | Зависит от TASK-034 (выбор пилотной точки), а не от своей фазы 4 |

## Таймлайн

```
неделя 1   │ ████ Phase 0
неделя 2-3 │ ████████ Phase 1
неделя 4   │ ████ Phase 2
           │ ████ Phase 3 (параллельно)
           │ ████ Phase 4 (параллельно)
           │ ████ Phase 5 (параллельно)
неделя 5   │ ██ Phase 6 → Sprint 0 старт
```

**Общая длительность:** ~5 календарных недель при наличии всех ответственных и параллельной работе Phase 3–5.

## Точки решения (Gates)

Каждая фаза заканчивается gate-решением, которое блокирует переход к следующей.

| Gate | Задача | Критерий перехода |
|------|--------|---------------------|
| **Phase 0 → 1** | TASK-006 | Запись в Decision Log: «Go / No-Go / Go with CR» |
| **Phase 1 → 2** | TASK-015 | Все 5 тех. документов в статусе `✅ Утверждён` |
| **Phase 2/3/4/5 → 6** | TASK-039 | Все артефакты Phase 0–5 закрыты, Health Check проекта зелёный |
| **Phase 6 → Sprint 0** | TASK-040 | Kick-off проведён, Sprint 0 запущен в трекере |

## Правила работы с бэклогом

1. **Формат задач** — единый, определён в [Task Standard](../templates/task-template.md). Каждая задача — отдельный файл `TASK-XXX.md`.
2. **Статусы задач** обновляются в самом файле задачи **и** в [Progress Tracker](progress.md).
3. **Закрытие задачи** — только после проверки всех критериев приёмки. Кто проверяет — указан в поле `A` (Подотчётный) карточки.
4. **Новые задачи** появляются через продолжение нумерации (TASK-041, 042, …) и регистрируются в [Progress Tracker](progress.md).
5. **Изменение скоупа задачи P0** — через [Change Request](../../05-governance/04-change-request-template.md).
6. **Блокировки** фиксируются в карточке задачи (статус `blocked`, причина, ID блокирующей задачи) и в [Progress Tracker](progress.md).
