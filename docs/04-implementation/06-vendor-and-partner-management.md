# Vendor & Partner Management

## 0. Цель документа

Зафиксировать, **как мы управляем внешними отношениями**, от которых зависит успех Orakul. Партнёры и вендоры — рычаг масштаба, но и источник ключевых рисков (R-01, R-05, R-09).

## 3. Стратегический партнёр: Kiepper

> Это **самый важный единичный партнёр**. Без него стратегия Orakul теряет дифференциацию.

### 3.1 Текущая роль партнёрства
- **Канал продаж:** Kiepper рекомендует Orakul своим клиентам.
- **Технический фундамент:** Orakul читает данные Kiepper по API.
- **Защита позиционирования:** партнёрство снижает вероятность конкуренции (R-01).

### 3.2 Условия (целевые)
| Параметр | Значение |
|----------|----------|
| Тип отношений | Меморандум о партнёрстве (не эксклюзив изначально) |
| Revenue share | 10–20% от первого года подписки клиента, пришедшего по каналу |
| API SLA | Партнёрский договор: ≥99.5% uptime API; уведомление об изменениях за 30 дней |
| Срок | 24 мес с авто-продлением; опт-аут за 90 дней |
| Cooperation | Совместные вебинары, кейсы, выступления на отраслевых событиях |

### 3.3 Что мы НЕ соглашаемся
- Эксклюзивная продажа только через Kiepper (мы должны иметь прямой канал).
- Передача данных клиентов обратно в Kiepper (только в виде черновиков заказов, по подтверждению пользователя).
- Совместная разработка фич без отдельного контракта.
- Любая привязка наших цен к ценам Kiepper.

### 3.4 Триггеры пересмотра / выхода

| Сигнал | Реакция |
|--------|---------|
| Kiepper выпускает свой ИИ-модуль (R-01) | Активация Plan B по GTM (см. ниже) |
| Партнёр-менеджер Kiepper меняется > 2 раз/год | Переоценка стабильности отношений |
| API breaking changes без 30-дневного уведомления | Эскалация до CEO Kiepper |
| Доля лидов от Kiepper <10% от плана 2 квартала подряд | Пересмотр распределения GTM-усилий |
| Kiepper рекомендует конкурента | Немедленная встреча и документ-разъяснение |

### 3.5 Plan B (если партнёрство срывается)

| Шаг | Действие | Сроки |
|-----|----------|-------|
| 1 | Усиление прямых продаж: +30% бюджет SDR/AE | ≤2 нед |
| 2 | Запуск Kiepper-независимого коннектора (нам не нужно партнёрство для интеграции с публичным API) | ≤8 нед |
| 3 | Ускорение партнёрств с консультантами и медиа | ≤4 нед |
| 4 | Коммуникация с существующими клиентами: «у нас всё в порядке» | ≤24 ч |
| 5 | Оценка возможности интеграции с другими POS/учётными системами (горизонт 2027+) | Решение в Decision Log |

## 4. Sub-processors (поставщики инфраструктуры)

### 4.1 Принципы
- **Минимум sub-processors.** Каждый дополнительный = поверхность атаки + точка отказа + строка в DPA.
- **Локализация по запросу клиента.** Для крупных сетей — возможность on-premise или РФ-only хостинг (опция Network).
- **Backup-провайдер.** Для критичной функциональности — план миграции на альтернативу за ≤30 дней.

### 4.2 Реестр sub-processors (актуализируется в DPA)

| Sub-processor | Категория | Назначение | Критичность | Backup |
|---------------|-----------|-------------|-------------|--------|
| Cloud-провайдер | Инфра | Хостинг приложения и БД | 🔴 Критическая | План миграции на альтернативный cloud, ≤30 дней |
| LLM-API провайдер | Алгоритмы | Генерация текстовых обоснований | 🟠 Высокая | Шаблонизатор fallback (без LLM) на критичные сценарии |
| Мониторинг (метрики, логи) | Операционная | Observability | 🟡 Средняя | Замена за ≤14 дней |
| Email-провайдер | Коммуникация | Транзакционные письма, дайджесты | 🟡 Средняя | Замена за ≤7 дней |
| Push-провайдер (мобайл) | Коммуникация | Уведомления управляющим | 🟡 Средняя | Замена за ≤7 дней |
| CRM (наш) | Внутренняя | Контакты, sales pipeline | 🟢 Низкая | Замена в плановом порядке |

### 4.3 Управление SLA
- Все sub-processors с критичностью ≥🟠 имеют контрактный SLA.
- При нарушении SLA — кредитный механизм (компенсация в виде скидки).
- Отчёт о SLA-нарушениях — ежемесячный, на стол TL.

### 4.4 Регулярный аудит
| Активность | Частота |
|-----------|---------|
| Полный обзор sub-processors | Ежегодно |
| Аудит безопасности (SOC2 / эквивалент) от провайдера | По выпуску отчёта (1–2 раза в год) |
| Pen-test нашей конфигурации на их инфраструктуре | Раз в полгода |
| Тренировка fail-over (для cloud-провайдера) | Раз в полгода |

## 5. Канальные партнёры (консультанты, маркетплейсы лидов)

### 5.1 Кто подходит
- Отраслевые консультанты по операционке общепита.
- Бухгалтерские/налоговые консультанты, обслуживающие сетевые рестораны.
- Образовательные платформы для шеф-поваров и управляющих.

### 5.2 Условия
| Параметр | Значение |
|----------|----------|
| Комиссия | 10% от первого года подписки клиента, пришедшего по каналу |
| Срок выплаты | После 90 дней удержания клиента (защита от churn-арбитража) |
| Право на использование бренда | Только в согласованных материалах |
| Эксклюзив | Не предоставляется |
| Минимум для активного партнёра | 2 сделки в полугодие, иначе перевод в «inactive» |

### 5.3 Что НЕ делаем
- Не платим комиссию за «холодный лид» (только за SQL → платящий).
- Не передаём партнёру данные клиента.
- Не передаём наш контракт на ведение партнёра.

## 6. Conflict of interest и красные линии

| Ситуация | Реакция |
|----------|---------|
| Партнёр одновременно работает с прямым конкурентом | Прозрачно. Внимательно следим за объёмом информации, который ему передаём. |
| Партнёр-консультант предлагает «купить лиды оптом» | Отказ. Это нарушение трассировки источника. |
| Sub-processor хочет использовать наши анонимизированные данные для своих моделей | Только с письменного согласия и в рамках DPA с клиентом. По умолчанию — нет. |
| Kiepper попросит «эксклюзив» на крупного клиента | Эскалация до CEO; решение через CR + Decision Log. |
| Лидер команды Orakul переходит к Kiepper (или наоборот) | Стандартные NDA + не-конкуренция; информирование через юриста. |
