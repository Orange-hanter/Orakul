# Стандарт оформления задачи

> **Назначение.** Единый формат для всех задач в [Delivery Backlog](../08-delivery-backlog.md) и в трекере (Jira/Linear). Стандарт обеспечивает однозначность, измеримость и трассируемость каждой задачи.

---

## 1. Обязательные поля задачи

| Поле | Что содержит | Пример |
|------|--------------|--------|
| **ID** | Уникальный идентификатор `TASK-XXX`, сквозная нумерация | `TASK-012` |
| **Название** | Императивная форма (что делаем). ≤80 символов | «Создать Integration Specification для Quick Resto API» |
| **Тип** | См. §2 | `Документ` |
| **Этап** | Номер этапа из Delivery Backlog (0–6) | `Phase 1` |
| **Приоритет** | См. §3 | `P0` |
| **Размер (Size)** | См. §4 | `M` (3–5 чел.-дней) |
| **Ответственный (R)** | Роль из [RACI](../02-raci-matrix.md), исполняет задачу | `Tech Lead` |
| **Подотчётный (A)** | Роль из RACI, отвечает за результат (1 человек) | `Владелец продукта` |
| **Описание** | 2–5 предложений: что и зачем. Связь с бизнес-результатом | — |
| **Критерии приёмки (DoD)** | Однозначные, проверяемые пункты. Каждый — `[ ]` | См. §5 |
| **Артефакт на выходе** | Конкретный файл / запись / решение / договор | `docs/08-technical/02-integration-spec.md` |
| **Зависимости** | ID других задач, которые должны быть завершены до старта | `TASK-001, TASK-003` |
| **Связанные документы** | Markdown-ссылки на BRD, PRD, User Stories, KPI | [BRD §6](../../02-business/01-business-requirements-document.md#6-функциональные-требования-бизнес-уровень) |
| **Связанные KPI** | Идентификаторы KPI из [KPI Framework](../../02-business/02-kpi-framework-and-dashboard.md) | `BIZ-05`, `OPS-05` |
| **Риски и блокеры** | Что может пойти не так и как митигируем | — |
| **Статус** | См. §6 | `pending` |

## 2. Типы задач

| Тип | Когда использовать | Артефакт |
|-----|--------------------|----------|
| **Документ** | Создание нового документа или раздела | `*.md` в репозитории |
| **Исследование (Spike)** | Технический разведочный эксперимент, ограниченный по времени | Отчёт + запись в Decision Log |
| **Решение** | Принять и зафиксировать архитектурное / продуктовое / коммерческое решение | Запись в [Decision Log](../../05-governance/05-decision-log.md) |
| **Юридика** | Договор, NDA, DPA, согласование с регулятором | Подписанный документ |
| **Инфра** | Развёртывание окружения, CI/CD, мониторинг | Работающий контур |
| **Тех. задача** | Конкретная инженерная задача (код) | PR в репозиторий продукта |
| **Коммуникация** | Совещание, согласование, демо | Протокол / sign-off |

---

## 3. Приоритеты

| Приоритет | Значение | Когда применять |
|-----------|----------|-----------------|
| **P0** | Блокер | Без задачи нельзя двигаться дальше по плану |
| **P1** | Важно | Нужна в текущем этапе, но не блокирует параллельную работу |
| **P2** | Желательно | Улучшение, можно отложить за пределы текущего этапа |
| **P3** | Опционально | Nice-to-have, если позволяет ресурс |

---

## 4. Размеры (Size)

Без участия команды разработки оценки — **предварительные**. Уточняются на Этапе 2 (Декомпозиция работ).

| Размер | Объём | Когда использовать |
|--------|-------|--------------------|
| **XS** | ≤1 чел.-день | Простое решение, короткая встреча, мелкая правка |
| **S** | 1–3 чел.-дня | Небольшой документ, простой спайк, шаблонная задача |
| **M** | 3–5 чел.-дней | Средний документ, договор, развёртывание окружения |
| **L** | 5–10 чел.-дней | Большой документ (TRD, SAD), сложный спайк, многосторонняя юридика |
| **XL** | >10 чел.-дней | Декомпозируется на более мелкие задачи |

## 5. Критерии приёмки (Definition of Done)

Критерии должны быть **проверяемыми без интерпретации**. Каждый пункт — чек-бокс.

### Правила формулирования

- ✅ «Документ содержит разделы 1–7, утверждён владельцем продукта, добавлен в `docs/`, ссылка в [README](../../../README.md)»
- ❌ «Документ написан качественно» (нельзя проверить)
- ✅ «API-ключ получен, ответ от `GET /storage/balances` возвращает ≥1 запись»
- ❌ «Доступ к API настроен» (неоднозначно)

### Базовые критерии для задач типа «Документ»

- [ ] Документ создан по согласованной структуре
- [ ] Все ссылки на связанные документы рабочие
- [ ] Документ прошёл ревью указанным согласующим
- [ ] Документ утверждён (статус `✅ Утверждён` в шапке)
- [ ] Ссылка на документ добавлена в [README](../../../README.md) и [Traceability Index](../../07-references/02-traceability-index.md)
- [ ] Создана запись в Changelog документа

### Базовые критерии для задач типа «Решение»

- [ ] Решение зафиксировано в [Decision Log](../../05-governance/05-decision-log.md) с датой и подписями
- [ ] Затронутые документы обновлены (через CR при необходимости)
- [ ] Стейкхолдеры уведомлены согласно [Communications Plan](../05-communications-plan.md)

## 6. Жизненный цикл задачи

```
pending → in_progress → review → done
                ↓
             blocked → (после разблокировки) → in_progress
```

| Статус | Когда устанавливать |
|--------|---------------------|
| `pending` | Создана, но ещё не начата. Все зависимости могут быть не закрыты |
| `in_progress` | Исполнитель приступил к работе |
| `blocked` | Заблокирована внешним фактором (с указанием причины и ожидаемой даты разблокировки) |
| `review` | Передана на ревью согласующему |
| `done` | Все критерии приёмки выполнены, артефакт принят |
| `cancelled` | Отменена с указанием причины (через CR при изменении скоупа) |

```markdown
| Поле | Значение |
|------|----------|
| Тип | Документ / Исследование / Решение / Юридика / Инфра / Тех. задача / Коммуникация |
| Этап | Phase X |
| Приоритет | P0 / P1 / P2 / P3 |
| Размер | XS / S / M / L / XL |
| Ответственный (R) | Роль |
| Подотчётный (A) | Роль |
| Зависимости | TASK-..., TASK-... |
| Связанные KPI | ID-... |
| Статус | pending |

**Описание.** Что делаем и зачем. Какую бизнес-задачу решает.

**Критерии приёмки:**
- [ ] Критерий 1 (проверяемый)
- [ ] Критерий 2
- [ ] ...

**Артефакт.** Конкретный путь к файлу / запись в системе.

**Связанные документы.** [Документ A](...), [Документ B](...).

**Риски и блокеры.** Что может пойти не так.
```

## 8. Правила работы с задачами

1. **Одна задача — один артефакт.** Если задача порождает несколько артефактов (например, документ + запись в Decision Log) — разбить на несколько задач.
2. **Не более одной задачи в `in_progress` на исполнителя.** Параллельность — между разными исполнителями.
3. **Блокировки фиксируются явно.** Если задача `blocked` — в карточке указывается причина и ID блокирующей задачи.
4. **Изменение скоупа задачи после старта** — через комментарий с обоснованием. Изменение принципиальных параметров (DoD, артефакт, ответственный) — через [Change Request](../../05-governance/04-change-request-template.md).
5. **Закрытие задачи** — только после проверки **всех** критериев приёмки согласующим.
