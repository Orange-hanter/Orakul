# Реестр стейкхолдеров

## 0. Зачем этот документ

- Обеспечить, что **никто из значимых сторон не упущен** ни на этапе принятия решений, ни в коммуникациях.
- Сделать явным **отношение каждой стороны к проекту** (поддерживает / нейтрален / сопротивляется) и **что с этим делать**.
- Привязать каждую сторону к чёткой **стратегии вовлечения**.

## 1. Категории стейкхолдеров

| Категория | Кто относится |
|-----------|---------------|
| **Внутренние — спонсоры** | CEO, Спонсор инвестора |
| **Внутренние — продуктовая команда** | PO, DL, TL, DES |
| **Внутренние — go-to-market** | SAL, MKT, CSM |
| **Внутренние — операционные** | PMO, Финдир, Юрист/Безопасник, HR |
| **Клиент — ЛПР** | Собственник, Опердир, Финконтролёр |
| **Клиент — пользователи** | Управляющий, Закупщик, Шеф-повар |
| **Клиент — IT** | IT-владелец Kiepper у клиента |
| **Внешние партнёры** | Kiepper (вендор), отраслевые консультанты, поставщики |
| **Внешние влияющие** | Регуляторы, отраслевые ассоциации, СМИ |

## 2. Матрица «Влияние × Заинтересованность»

```
Влияние ↑
   Высокое  │  ☆ Управлять плотно        │  ☆ Управлять плотно
            │  (Опердир клиента)         │  (Спонсор / CEO,
            │  (IT-владелец Kiepper)     │   Собственник клиента,
            │                            │   Финконтролёр клиента)
            │                            │
   Среднее  │  ☆ Удовлетворять           │  ☆ Информировать /
            │  (Регуляторы)              │     консультировать
            │  (Отраслевая ассоциация)   │  (PO, CSM, DL)
            │                            │  (Управляющий, Шеф-повар)
            │                            │
   Низкое   │  ☆ Мониторить              │  ☆ Информировать
            │  (СМИ, общая аудитория)    │  (Закупщик пилотных точек,
            │                            │   линейный персонал)
            └────────────────────────────┴───────────────────────────→
                Низкая                       Высокая    Заинтересованность
```

> Стратегия в каждом квадранте:
> - **Управлять плотно (Manage closely):** регулярные встречи, активное вовлечение в решения.
> - **Удовлетворять (Keep satisfied):** не перегружать деталями, но обеспечивать соответствие требованиям.
> - **Информировать (Keep informed):** регулярная рассылка, без обязательного двустороннего диалога.
> - **Мониторить (Monitor):** минимум контактов, но отслеживать сигналы.

## 4. Профили ключевых стейкхолдеров (deep-dive)

> Полные профили ведутся для всех stakeholder'ов с влиянием = «Высокое» в режиме «Manage closely».

### Шаблон профиля

```
ID: S-01 (CEO / Спонсор)
Должность: Генеральный директор Orakul
Зона ответственности: Стратегия, финансирование, партнёрство Kiepper
Что для него «успех» проекта: 15 логотипов / 50 точек, NRR ≥110%, ROI клиентов подтверждён
Что для него «провал»: Kiepper-партнёрство не подписано + Pilot→Paid <60%
Личные предпочтения: предпочитает короткие письма, нелюбит перегруженные слайды
Триггеры эскалации: любой G-gate не пройден; уход >2 ключевых сотрудников
Канал быстрой связи: Telegram, ответ в течение 2ч в рабочее время
Регулярность встреч: еженедельный 1:1 (понедельник 10:00), MBR (1-й рабочий день месяца)
```

> Аналогичный профиль ведётся для C-01 (собственник клиента), C-02 (опердир), C-07 (IT Kiepper) и E-01 (партнёр-менеджер Kiepper). Детали — во внутреннем CRM, не публикуются.

## 5. Переходы по отношению

| Текущее → Желаемое | Что делает разницу |
|---------------------|---------------------|
| Сопротивляется → Нейтрален | Понять причину сопротивления (страх, прошлый опыт, информационный пробел). Не убеждать — слушать. См. [Change Management Plan](../04-implementation/03-change-management-plan.md). |
| Нейтрален → Поддерживает | Дать персональную ценность (что меняется в моей работе). Демо на их данных, не на абстракциях. |
| Поддерживает → Чемпион | Включить в кейс, дать публичность, признать вклад. |

## 6. Регламент работы с реестром

| Активность | Частота | Ответственный |
|-----------|---------|---------------|
| Идентификация нового стейкхолдера | По мере появления | Любой → PMO |
| Заполнение профиля | ≤7 рабочих дней с идентификации | CSM (для клиента) / PMO (для прочих) |
| Переоценка отношения | Каждые 4 недели на пилоте; ежемесячно после запуска | CSM + PMO |
| Полный пересмотр реестра | Ежеквартально (QBR) | PMO |
