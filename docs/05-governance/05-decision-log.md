# Decision Log

## 0. Назначение

Один файл, в который **попадают все значимые решения проекта**. Не для бухгалтерии — для того, чтобы:
- через 6 месяцев можно было вспомнить, **почему** мы так сделали (а не только что);
- новый член команды за час понял, какие развилки уже пройдены;
- при споре «договаривались иначе» был источник правды.

> **Правило:** если решение не в этом журнале — оно не принято. Что бы ни происходило в чате.

### DL-2026-002 — North Star: ARAR

**Дата:** 2026-05-07  
**Категория:** Strategic

**Вопрос.** Какая метрика должна быть единственной, по которой компания оценивает себя?

**Опции.**
- A) MRR / ARR (классический SaaS).
- B) Точность прогноза (MAPE).
- C) ARAR — % принятых/скорректированных рекомендаций.
- D) Снижение списаний у клиентов.

**Решение.** C — ARAR.

**Почему:**
1. Adoption — leading indicator всех остальных метрик. Без adoption нет экономии, нет retention, нет MRR.
2. Защищает от «галочного» использования (counter-metric Override Rate).
3. Простая в коммуникации: одна цифра, понятная всем ролям.

**От чего отказались:** A — financial outcome, не leading. B — техническая, ничего не говорит о ценности. D — слишком зависит от клиента, не от нашей команды.

**Последствия:**
- В Vision §9 зафиксировано.
- В KPI Framework — NSM-01.
- Все продуктовые KPI выводятся как support-metrics к ARAR.
- **Review trigger:** конец Q4 2026 — пересматриваем, остаётся ли ARAR North Star или продукт перерос.

**Подписи:** CEO, PO, PMO.

### DL-2026-003 — Принцип «Recommend, don't act»

**Дата:** 2026-05-07  
**Категория:** Product

**Вопрос.** Должен ли Orakul автоматически выполнять действия (создавать заказ, менять рецептуру) при высокой уверенности?

**Опции.**
- A) Да — при confidence >X% запускаем без подтверждения.
- B) Нет — всегда требуется явное подтверждение пользователя.
- C) Гибрид — для мелких операций авто, для крупных — ручное.

**Решение.** B — всегда подтверждение.

**Почему:**
1. Доверие пользователя строится только на контроле; авто = подрыв доверия.
2. Юридическая ответственность за заказ всегда у человека.
3. Авто-действия требуют отдельного DPA-блока с клиентом, что замедлит сделки.
4. Принцип объяснимости теряет смысл, если действие уже произошло до объяснения.

**От чего отказались:** A — слишком высокий риск. C — сложность тарифа и UX.

**Последствия:**
- Зафиксировано в [PRD §3](../03-product/01-product-requirements-document.md).
- Любое будущее исключение = отдельный CR с обоснованием и согласием клиента.
- **Review trigger:** конец 2027, после 24 месяцев работы продукта — пересмотр в свете накопленной статистики.

**Подписи:** CEO, PO, Безопасник.

### DL-2026-004 — Тарифная сетка

**Дата:** 2026-05-07  
**Категория:** Commercial

**Вопрос.** Какую тарифную модель использовать на старте?

**Опции.**
- A) Фиксированная подписка (одна цена).
- B) Per location, 3 тарифа (Starter / Growth / Network).
- C) Per user.
- D) % от экономии клиента (revenue share).

**Решение.** B — per location, 3 тарифа.

**Почему:**
1. Per location устраняет трение «не давать доступ».
2. 3 тарифа покрывают сегменты P0/P1.
3. Revenue share (D) красив в теории, но создаёт спор о baseline и методике расчёта; убивает скорость продажи.

**От чего отказались:** A — не масштабируется на сети 20+. C — анти-adoption. D — слишком сложный пилот.

**Последствия:**
- Зафиксировано в [Pricing](../06-financial/02-pricing-and-commercial-model.md).
- Pilot-цена = 50% Growth.
- **Review trigger:** ежегодно (1 декабря).

**Подписи:** CEO, Sales Lead, Финдир.

### DL-2026-005 — Pilot Program

**Дата:** 2026-05-07  
**Категория:** Commercial / Process

**Вопрос.** Каковы условия пилота?

**Опции.**
- A) Бесплатно на 4 недели.
- B) 50% от Growth-цены, 6 недель, с гарантией возврата при недостижении KPI.
- C) Полная цена, 12 недель.

**Решение.** B.

**Почему:**
1. Платный пилот = серьёзность намерений клиента (skin in the game).
2. 6 недель = достаточно для baseline + замера эффекта.
3. Гарантия возврата = отрицание возражения «а вдруг не сработает».

**От чего отказались:** A — клиенты не ценят бесплатное; низкая мотивация дать ресурсы. C — слишком долго, цикл сделки удлиняется.

**Последствия:**
- Зафиксировано в [GTM §6](../01-strategy/03-go-to-market.md), [Pricing §3](../06-financial/02-pricing-and-commercial-model.md).
- **Review trigger:** после 5 пилотов (Q3 2026) — анализ конверсии и пересмотр условий.

**Подписи:** CEO, Sales Lead.

### DL-2026-006 — Data Audit как gating

**Дата:** 2026-05-07  
**Категория:** Process / Risk Mitigation

**Вопрос.** Может ли клиент стартовать пилот, не пройдя Data Audit?

**Опции.**
- A) Да, разберёмся по ходу.
- B) Нет, без подписанного Audit Report переход E1→E2 заблокирован.

**Решение.** B.

**Почему:**
1. Риск R-04 (низкое качество данных) — Severity 20 🔴, самый высокий в реестре.
2. Запуск без аудита = гарантированные «галлюцинации» ИИ → провал пилота → удар по репутации.
3. Аудит = часть нашей ценности, не накладные расходы.

**От чего отказались:** A — гарантированно создаёт инциденты, которые мы не можем контролировать.

**Последствия:**
- В каждом контракте есть пункт «E1 проходится по [Data Audit Checklist](../04-implementation/04-data-audit-checklist-kiepper.md)».
- Если клиент отказывается — мы не подписываем (это red flag по ICP).
- **Review trigger:** после 10 клиентов — анализ «не слишком ли строгий» аудит.

**Подписи:** CEO, PMO, DL.

### DL-2026-008 — Миграция на Flutter отложена до G3

**Дата:** 2026-05-21
**Категория:** Product / Tech

**Вопрос.** Стоит ли уже сейчас мигрировать клиентское приложение с React+Vite (PWA) на Flutter для будущей кросс-платформенности?

**Опции.**
- A) Мигрировать сейчас (до завершения R2) — фронт переписывается с нуля.
- B) Мигрировать постепенно, начиная с новых модулей (каталог поставщиков на Flutter, остальное на React).
- C) Остаться на React+Vite (PWA) до Gate G3; пересмотреть тогда.

**Решение.** C — оставаться на React+Vite до Gate G3.

**Почему:**
1. Текущий стек работает: пилот на iPad Safari, жалоб от пользователей нет ([Pilot App Deployment](../08-technical/05-pilot-app-deployment.md)).
2. Фронт ещё несколько раз перепишется до product-market fit. Миграция сейчас = переписывать заведомо временный код.
3. Реальной кросс-платформенной потребности нет: персоны из [PRD §4](../03-product/01-product-requirements-document.md) покрываются PWA. App Store-витрина нужна только под мобильные push (можно через [Telegram Bot](../08-technical/04-telegram-notification-spec.md)).
4. Цена: переписать каталог + заявки на Flutter = 6–8 недель; на React = 4 недели + начало финаналитики. Каждый месяц задержки V2 = недополученные 4.2 млн ₽/год экономии у клиента.
5. Подготовка к будущей миграции уже сделана: фронт изолирован от бизнес-логики через REST API.

**От чего отказались:**
- A — преждевременная оптимизация; убивает скорость R2.
- B — параллельный стек (React + Flutter) удваивает поддержку без выигрыша.

**Последствия:**
- Все новые фичи R2 (каталог, заявки, финаналитика) — на React+Vite.
- Дизайн-токены оформляются в нейтральной нотации (CSS variables → совместимо с Flutter ThemeData при будущем переезде).
- Кандидаты на будущую миграцию: Flutter / Tauri / PWA+Capacitor — выбор откладывается.
- **Review trigger:** после Gate G3 (конец Q3 2026, ≥3 клиента, ≥10 точек) или при первом клиенте, явно требующем нативное приложение.

**Подписи:** Tech Lead.

### DL-2026-009 — Каталог поставщиков и заявки переносятся в R2

**Дата:** 2026-05-21
**Категория:** Product / Timeline
**Связанный CR:** [CR-2026-001](cr-2026-001-supplier-catalog-to-r2.md)

**Вопрос.** Когда поставлять US-09 (каталог поставщиков), US-10 (поиск аналогов) и US-11 (мастер заявок) — в R2 (Q2) или R3 (Q3, как изначально планировалось для US-04)?

**Опции.**
- A) R3 (Q3) — как было запланировано.
- B) Только US-09 в R2; US-10/11 в R3.
- C) Все три (US-09/10/11) в R2, в сокращённом MVP-объёме.

**Решение.** C — все три в R2, MVP scope.

**Почему:**
1. Эта функциональность — единственный канал Value Driver V2 в [Business Case §2](../06-financial/01-business-case-and-roi.md). Без неё ROI клиента «недосчитывается» на 60–70% в первый год.

3. US-09 без US-10/11 = «справочник», не workflow. Фрагментация убивает «вау»-эффект для собственника.
4. MVP-формат (ручной ввод цен, точный поиск, упрощённый мастер заявки) укладывается в 6 недель без расширения команды.

**От чего отказались:**
- A — упущенная экономия у клиента 4.2 млн ₽/год/точка не доказывается, paid-конверсия под риском.
- B — нет workflow «от инсайта к действию».

**Последствия:**
- [Roadmap R2 и R3](../04-implementation/01-roadmap.md) обновлён.
- [User Stories](../03-product/02-user-stories.md) — приоритет US-09/10/11 поднят с P1 до P0.
- [PRD §5](../03-product/01-product-requirements-document.md) — таблица модулей обновлена.
- Импорт xlsx, рейтинги поставщиков, fuzzy-поиск аналогов — остаются в R3 как «расширение каталога».
- **Review trigger:** середина июня 2026 — контрольная точка по rollback-плану CR (если объём работ превышает оценку >30%, US-11 переносится обратно в R3).

**Подписи:** Tech Lead, PO (требуется подтверждение Спонсора и PMO).

- **F04 + F06 Telegram алёрты** — повышение цены поставщика > 5% / 1 BYN автоматически шлёт алёрт; рассчитывается влияние на маржу блюд.
- **F05 Утренний P&L digest** — собственник получает вчерашние выручку/FC%/EBITDA в Telegram в 09:00 без логина.
- **F07 Onboarding checklist** — первый запуск показывает 4-шаговый чек-лист готовности данных.

**Метрики:**
- 116 unit-тестов зелёных (старт: 72).
- 5 production-релизов через rsync + systemctl, без downtime.
- Прод бандл: 84 KB gzip main + 163 KB gzip xlsx (lazy).
- Новые типы записей: `dish_sale` (venue-scoped, миграция отработала на проде).

**Что разблокировано для Sprint 2:**
- Forecast (AI01) имеет структуру продаж по дням → можно считать MAPE.
- ARAR-трекинг (AI04) может опираться на user-actions поверх существующих карточек.
- Anomaly detection (AI05) переиспользует writeoffControl-логику.

**От чего отказались:** ни от чего — все 8 задач выполнены.

**Последствия:**
- Sprint 2 «Первый ИИ-слой» открыт без блокеров.
- Telegram-инфраструктура готова для дальнейших алёртов в Sprint 2/3.
- Гейтинг Sprint 1 → Sprint 2 пройден (см. roadmap §«Гейтинг»).

**Review trigger:** при первом подписанном paying-клиенте — пересмотреть KPI Sprint 1 на основании реальной обратной связи.

### DL-2026-010 — RBAC поэтапно, начиная с multi-venue

**Дата:** 2026-05-22
**Категория:** Product / Tech

**Вопрос.** Что реализуем в качестве «RBAC» под US-15 (сравнение точек) — полную спецификацию из [08-rbac-multiuser-spec.md](../08-technical/08-rbac-multiuser-spec.md) или часть?

**Опции (4 независимых слоя RBAC):**
- L1: Multi-user login (учётные записи вместо общего APP_PASSWORD)
- L2: Multi-venue scope (venueId на записях + селектор точки)
- L3: Привязка user ↔ venues
- L4: Permission matrix по ролям

**Решение.** Только L2 (multi-venue) в текущей итерации.

**Почему:**
1. US-15 требует только multi-venue. RBAC слои L1/L3/L4 — для сценариев с несколькими сотрудниками внутри клиента, что не покрывает текущий пилот.
2. На пилоте «собственник = единственный пользователь» добавление учётных записей удваивает поверхность работы без бизнес-ценности.
3. L2 даёт минимальную миграцию данных (venueId на 8 типов записей) и одну новую сущность venue. ~1–2 дня.
4. Полная спека [08-rbac-multiuser-spec.md](../08-technical/08-rbac-multiuser-spec.md) остаётся как guidance — её реализация поэтапна, в R3+ при появлении первого клиента с несколькими сотрудниками.

**От чего отказались:**
- Полная RBAC сейчас — 10–14 дней работ, откладывает US-15 на 2 недели без видимой пользы для пилота.
- Только L1 без L2 — не разблокирует US-15.

**Последствия:**
- Сервер: миграция `migrateMultiVenue()` создаёт «Точка 1» при первом запуске и проставляет `venueId` существующим записям. Идемпотентна.
- Сервер: safety net в `POST /api/records` подставляет `venueId` default-точки, если клиент не прислал.
- Фронтенд: селектор точки в шапке (`📍 ...`), `effectiveVenueId` в `Main.jsx`, фильтр venue-scoped записей в ctx, авто-инъекция `venueId` при `onCreate`.
- Аутентификация: общий APP_PASSWORD без изменений.
- **Known limitation:** Telegram-дайджест пока не фильтрует продукты по venueId — чаты получают агрегированный отчёт по всем точкам. Исправление — в L3 (когда чаты надо разделять между ролями).
- **Review trigger:** при подписании первого клиента с ≥2 сотрудниками → пересмотр L1/L4.

**Подписи:** Tech Lead.

## 6. Анти-паттерны

| Анти-паттерн | Почему плохо |
|--------------|---------------|
| Решение «in chat», не в логе | Через 3 месяца никто не помнит |
| Запись только «что решили», без «почему» | Бесполезно при пересмотре |
| Длинные карточки на тривиальные вопросы | Шум; теряются важные решения |
| Не фиксируется отказ от чего-то | Через год кто-то заново предлагает то же самое |
| Карточка без review trigger там, где он напрашивается | Решения «зависают» в неактуальных формах |
