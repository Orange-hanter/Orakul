# Data Governance

## 0. Цель документа

Зафиксировать политики и процессы работы с данными клиента в Orakul. Документ обязателен для:
- подписания DPA с клиентом;
- прохождения аудита безопасности у крупных клиентов;
- внутренней дисциплины команды;
- защиты от рисков R-06 (утечка) и R-07 (комплаенс).

---

## 1. Принципы (Data Principles)

| Принцип | Что значит |
|---------|------------|
| **Data minimization** | Хранить только то, что необходимо для рекомендаций. Никаких персональных данных гостей точек, никаких ФИО водителей поставщика и т.п. |
| **Customer owns data** | Все мастер-данные принадлежат заказчику. Orakul — обработчик, не оператор. |
| **No write-back to master** | Orakul **никогда** не пишет в мастер-данные Kiepper (рецептуры, цены, справочники). Только транзакционные сущности (черновики заказов) и только после явного подтверждения. |
| **Encrypt by default** | Все данные шифруются in-transit (TLS 1.2+) и at-rest (AES-256). |
| **Least privilege** | Каждая роль (внутри клиента и внутри команды Orakul) видит только то, что нужно для её работы. |
| **Audit everything that matters** | Любой доступ к данным клиента, любое изменение конфигурации, любой административный экспорт — в журнале ≥3 года. |
| **Separation of duties** | Создание данных (рецептур) и их утверждение (заказов) разнесены по ролям. См. [PRD §7](../03-product/01-product-requirements-document.md). |
| **Right to delete** | По запросу клиента — полное удаление данных в течение 30 дней с подтверждением. |

## 2. Каталог данных (Data Inventory)

### 2.1 Категории данных

| Категория | Примеры | Источник | Чувствительность |
|-----------|---------|----------|------------------|
| **Мастер-справочники** | SKU, рецептуры, поставщики, точки | Kiepper | Средняя (коммерческая тайна) |
| **Транзакционные** | Продажи, чеки, остатки, списания, закупки | Kiepper | Средняя |
| **Расчётные (наши)** | Прогнозы, рекомендации, аномалии | Orakul | Средняя |
| **Поведенческие (наши)** | Логины, действия пользователей, клики на рекомендации | Orakul | Низкая (внутренняя) |
| **Финансовые KPI** | Маржа, food cost, ROI | Производное | **Высокая** (коммерческая тайна) |
| **Идентифицирующие персонал** | ФИО управляющих/закупщиков (учётные записи) | Клиент | **Высокая (PII)** |
| **Гости заведения** | (НЕ обрабатываем) | — | — |
| **Платёжные данные клиента** | (НЕ обрабатываем) | Биллинг отдельно | — |

### 2.2 Что мы НЕ собираем
- ФИО, телефоны, email посетителей точек.
- Платёжные данные посетителей.
- Видео/аудио с камер.
- Данные сотрудников, не связанные с их рабочими действиями (паспорт, зарплата, табель).

> Если в будущем потребуется (например, привязка к поведению гостей) — это новая категория, требующая отдельного DPA, новой оценки рисков и согласия клиента.

## 3. Жизненный цикл данных

### 3.1 Сбор
- Только через утверждённый интеграционный канал с Kiepper.
- Список полей задокументирован и согласован с клиентом в DPA.
- Любое расширение списка полей = CR.

### 3.2 Хранение
| Категория | Срок хранения | Где хранится |
|-----------|---------------|---------------|
| Мастер-справочники | На время действия контракта + 30 дней grace | Production DB, шифрование |
| Транзакционные | Активные 24 мес, далее в архив до 60 мес | Production DB → Cold storage |
| Audit-trail | 36 мес минимум | Append-only log |
| Логи приложения | 90 дней (после анонимизации) | Логи |
| Backup | 30 дней с rolling rotation | Изолированный бакет, шифрование |

### 3.3 Использование
- **В пределах контура клиента.** Данные одной сети **никогда** не используются для обучения моделей в пользу другой сети без явного письменного согласия.
- **Anonymized aggregates допустимы** для бенчмарков отрасли (например, «средняя маржа в сетевом ресторане»), только если данные >10 клиентов и невозможно деанонимизировать.

### 3.4 Передача третьим сторонам
- **По умолчанию — нет.** Никаких данных не передаётся партнёрам, маркетингу, аналитикам без явного согласия клиента.
- **Sub-processors** (cloud-провайдер, LLM-API, мониторинг) указаны в DPA с описанием того, какие данные и зачем им идут.
- **При смене sub-processor** — уведомление клиента за 30 дней с правом отказа от продления.

### 3.5 Удаление
- По запросу клиента — в течение 30 дней.
- При расторжении контракта — автоматически через 30 дней с подтверждением.
- Cold backup также удаляется в течение 60 дней.
- Подтверждение об удалении — письменное, с подписью CTO/SEC.

## 4. Доступы (Access Control)

### 4.1 Внутренние (команда Orakul)

| Роль | Боевые данные клиента | Логи | Конфигурация | Анонимизированные агрегаты |
|------|------------------------|------|---------------|------------------------------|
| CEO | ❌ | ❌ | C | ✅ |
| PO | ❌ | ✅ (read) | ✅ | ✅ |
| DL | ✅ (только обезличенные / тестовые) | ✅ | ✅ | ✅ |
| TL / DevOps | ⚠ (только на инцидент, через JIT-доступ) | ✅ | ✅ | ✅ |
| CSM (назначенный на клиента) | ✅ (read только своих клиентов) | ⚠ | ❌ | ✅ |
| SEC | ✅ (на инцидент, через JIT) | ✅ | ✅ | ✅ |
| Маркетинг | ❌ | ❌ | ❌ | ✅ |

> **JIT-доступ** (just-in-time) — доступ выдаётся на конкретный инцидент, на ограниченное время (≤4 ч), с фиксацией причины. Все JIT-доступы пересматриваются ежеквартально.

### 4.2 Со стороны клиента
| Роль | Что видит | Полный список — в [PRD §7](../03-product/01-product-requirements-document.md) |
|------|-----------|--|
| Собственник | Все точки, все KPI | |
| Опердир | Все точки, все KPI | |
| Финконтролёр | Все точки, финансовые KPI | |
| Управляющий | Только своя точка | |
| Закупщик | Закупочный пул | |
| Шеф-повар | Рецептуры, food cost | |

### 4.3 Принципы аутентификации
- **Двухфакторная** для всех ролей с правом записи (опердир, управляющий, закупщик, шеф-повар, финконтролёр).
- **SSO** (single sign-on) для крупных клиентов — опция Network-тарифа.
- **Сессия** ≤8 ч; auto-logout после 30 мин неактивности.
- **Failed-login lockout** после 10 попыток за 15 мин.

## 5. Журналирование и аудит

### 5.1 Что обязательно логируется
- Все логины и попытки входа.
- Все экспорты (кто, когда, что, в каком формате).
- Все изменения настроек (роли, пороги, RBAC).
- Все события рекомендаций (сгенерирована, показана, действие пользователя — см. [PRD §6](../03-product/01-product-requirements-document.md)).
- Все JIT-доступы команды Orakul.
- Все изменения схемы данных в Orakul.

### 5.2 Доступ к логам
- Клиент имеет доступ к **аудит-журналу своих данных** через интерфейс (read-only).
- Логи команды Orakul (JIT, deploy и т.д.) — внутренние, доступ — SEC + TL.
- Внешний аудит — по контракту, через PMO.

### 5.3 Защита логов
- Append-only хранилище.
- Криптографическая привязка к timestamp.
- Backup отдельно от основной БД.
- Невозможность удаления записей даже под root (только архивирование).

## 6. Реакция на инциденты безопасности

### 6.1 Категории инцидентов

| Класс | Пример | Реакция |
|-------|--------|---------|
| **C0 — катастрофа** | Утечка данных нескольких клиентов | План кризис-коммуникации (Comms §6); уведомление РКН ≤72 ч; разбор с клиентами |
| **C1 — критический** | Несанкционированный доступ к одному клиенту | Заблокировать доступ ≤1 ч; уведомить клиента ≤24 ч; post-mortem ≤7 дней |
| **C2 — серьёзный** | Внутренний нарушитель политики (без утечки) | Внутреннее расследование; дисциплинарные меры; ужесточение процедур |
| **C3 — мелкий** | Failed-login лавина (брутфорс) | Авто-блокировка IP; отчёт в SOC сводно |

### 6.2 Регламент реакции
1. **Час 0:** обнаружение → SEC + TL немедленно. CEO уведомлён ≤30 мин.
2. **Час 1:** оценка масштаба, изоляция, начало расследования.
3. **Час 2–24:** митигация, восстановление, уведомление клиентов (через CSM + CEO).
4. **≤72 ч:** уведомление регуляторов (если применимо по ФЗ-152 / GDPR).
5. **≤7 дней:** post-mortem, план изменений процессов.
6. **≤30 дней:** изменения процессов внедрены, проверены SEC, отчёт клиентам.

## 7. Регуляторный комплаенс

| Регулятор / Норма | Применимость | Что мы делаем |
|--------------------|---------------|----------------|
| ФЗ-152 («О персональных данных», РФ) | Да, при работе на территории РФ | Локализация хранения PII в РФ; уведомление РКН при инциденте; согласия в DPA |
| GDPR (ЕС) | Только при работе в ЕС | DPA, DPIA при необходимости |
| Отраслевые требования общепита (СанПиН и пр.) | Косвенно (через клиента) | Поддержка экспортов и аудит-следа |
| Налоговые требования (хранение первичных) | Через клиента (Kiepper хранит) | Не подменяем Kiepper |

> Юридическое сопровождение — обязательно. Изменения регуляторики отслеживаются ежемесячно (юрист). При изменениях >минорных — CR на обновление процессов.

## 9. DPIA (Data Protection Impact Assessment)

> Краткий реестр оценок воздействия на данные. Полные DPIA — внутренние.

| Активность | Уровень риска | Ключевая митигация |
|-----------|----------------|---------------------|
| Хранение продаж и рецептур | Низкий | Шифрование, RBAC |
| Хранение учётных записей пользователей | Средний | 2FA, lockout, SSO option |
| Использование LLM для обоснований | Средний | Только обезличенные данные передаются; прокси-фильтр |
| Анонимизированные агрегаты для бенчмарков | Низкий | Минимум 10 клиентов в выборке |
| Передача данных при кризисе на партнёра-Kiepper | Высокий — НЕ происходит | Запрет в DPA |
