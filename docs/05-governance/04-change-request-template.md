# Change Request Template

## 0. Когда использовать CR

Любое изменение, которое:
- меняет периметр (in/out of scope) проекта;
- меняет сроки этапов или Roadmap;
- меняет KPI или их пороги (после старта baseline);
- меняет бюджет или коммерческие условия;
- меняет роли и ответственность (RACI);
- меняет принципы продукта или политики Data Governance;
- активирует план B (например, plan B по партнёрству Kiepper).

**Когда CR НЕ нужен:** мелкие правки внутри уже утверждённого scope (опечатки, переформулировки, тестовые корректировки), которые не меняют ожидания заказчика.

## 2. Матрица обязательных согласований

| Категория CR | Обязательные подписи |
|---------------|----------------------|
| Scope клиента (in/out of scope) | Спонсор клиента + Опердир клиента + PO + PMO |
| Timeline (Roadmap, gates G1..G4) | Спонсор + PO + PMO + (для клиента — Опердир) |
| KPI / метрики | Финдир + PO + (для клиента — Финконтролёр) |
| Budget / pricing | Спонсор + Финдир + (для контракта — Спонсор клиента) |
| RACI / роли | PMO + Спонсор + (затронутые роли) |
| Product principles | PO + Спонсор + Безопасник (если касается доступа/комплаенса) |
| Data Governance / Security | Безопасник + Юрист + Спонсор + IT-владелец Kiepper |
| Активация Plan B (партнёрство) | Спонсор + CEO + Sales Lead |

---

## 3. SLA на обработку CR

| Категория | Время от подачи до решения |
|-----------|------------------------------|
| Cтандартный (scope, timeline, RACI) | ≤ 5 рабочих дней |
| Срочный (триггер риска, инцидент) | ≤ 1 рабочий день |
| Стратегический (vision, GTM) | ≤ 10 рабочих дней с обязательной встречей управляющего комитета |
| Низкий приоритет (улучшение процесса) | ≤ 15 рабочих дней |

---

## 4. Цикл жизни CR

```
   Draft  →  On Review  →  Approved        →  Implemented
                  ↓
              Rejected   →  В архив с обоснованием
```

| Состояние | Что значит |
|-----------|------------|
| Draft | Заполнен инициатором, не отправлен на согласование |
| On Review | Передан матричным согласующим |
| Approved | Все подписи собраны → автоматически фиксируется в Decision Log |
| Rejected | Не утверждён; обоснование в Decision Log |
| Implemented | Изменения внесены в документы; пост-факт оценка зафиксирована |

---

## 5. Правила нумерации

`CR-YYYY-NN`, где:
- `YYYY` — год;
- `NN` — порядковый номер в году, сквозной (2-значный, ведущие нули).

Пример: `CR-2026-007`.

---

## 6. Где хранится

- Активные CR — в системе управления проектами (CRM/Jira/PMO-tool).
- Утверждённые CR — обязательно зафиксированы в [Decision Log](05-decision-log.md).
- Внесённые в документы изменения — отражены в Changelog соответствующего документа.

---

## 7. Что НЕ является CR

| Активность | Почему |
|-----------|--------|
| Опечатка / форматирование | Минорный patch; обновляется напрямую с инкрементом patch-версии |
| Бэклог-задачи в текущем спринте | Это backlog grooming, не CR |
| Внутренние процессы команды Orakul (без влияния на клиента) | Внутреннее решение в Decision Log без CR |
| Реакция на инцидент в реальном времени | Сначала действуем, CR оформляется ретроспективно как «emergency CR» |
