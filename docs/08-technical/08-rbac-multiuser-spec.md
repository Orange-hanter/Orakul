# Спецификация многопользовательского доступа и RBAC

## 1. Назначение

Документ описывает модель управления пользователями, ролями и правами доступа в системе Orakul. Цель — обеспечить безопасную совместную работу нескольких сотрудников одной или нескольких заведений под единой подпиской с чёткой границей видимости данных.

---

## 2. Иерархия сущностей

```
Organization (организация / сеть)
└── Venue (заведение / точка)
    └── User ← Role (роль)
                └── Permissions (права)
```

| Уровень | Описание |
|---------|----------|
| **Organization** | Юридическое лицо или группа (сеть). Биллинг привязан сюда. |
| **Venue** | Конкретная точка общепита внутри организации. |
| **User** | Физический сотрудник. Может иметь разные роли в разных точках. |
| **Role** | Набор разрешений. Назначается пользователю в контексте конкретной точки или всей организации. |

**Пример:** Иванов И.И. — `manager` в «Кофейне №1» и `viewer` в «Кофейне №2»; Петров П.П. — `ops_director` на уровне организации (видит все точки).

---

## 3. Предустановленные роли

### 3.1 Роли уровня организации

| Роль | ID | Описание |
|------|-----|----------|
| **Owner** | `owner` | Владелец / собственник. Полные права. Создаёт организацию, управляет биллингом. |
| **Ops Director** | `ops_director` | Операционный директор. Доступ ко всем точкам, но без управления биллингом и пользователями орг-уровня. |

### 3.2 Роли уровня точки

| Роль | ID | Основная задача |
|------|-----|----------------|
| **Manager** | `manager` | Управляющий точкой. Полный функционал точки. |
| **Purchaser** | `purchaser` | Закупщик / товаровед. Акцент на закупках и поставщиках. |
| **Chef** | `chef` | Шеф-повар / технолог. Рецептуры, меню, факт vs норма. |
| **Finance Controller** | `fin_controller` | Финконтролёр. Аналитика, отчёты, аудит. Только чтение по оперативным данным. |
| **Staff** | `staff` | Рядовой сотрудник. Очень ограниченный доступ (например, отметить приход товара). |
| **Viewer** | `viewer` | Только просмотр дашборда. Без возможности изменений. |

## 5. Управление пользователями (Admin UI)

### 5.1 Экран «Команда»

```
┌──────────────────────────────────────────────────────────────────┐
│  Команда                             [+ Пригласить сотрудника]   │
│  ──────────────────────────────────────────────────────────────  │
│  Фильтр: [Все точки ▾]  [Все роли ▾]  [Статус: Активные ▾]      │
│                                                                  │
│  Имя                   Email                 Роль       Точки    │
│  ──────────────────────────────────────────────────────────────  │
│  Иванов Иван          ivan@cafe.by          Manager    Кофейня 1 │
│  Петрова Анна         anna@cafe.by          Purchaser  Кофейня 1 │
│                                                                  │
│                                                         Кофейня 2│
│  Козлов Алексей       alex@cafe.by          Chef       Кофейня 1 │
│  Морозова Елена       elena@cafe.by         Fin.Ctrl   Все точки │
│  ──────────────────────────────────────────────────────────────  │
│  Ожидают принятия приглашения: 2                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 5.2 Процесс приглашения

```
1. Owner/Manager нажимает «Пригласить сотрудника»
2. Вводит email и выбирает роль + точки (или «Все точки»)
3. Система отправляет письмо с ссылкой (TTL: 72 часа)
4. Сотрудник переходит по ссылке, регистрируется (или входит если уже есть аккаунт)
5. Автоматически получает указанную роль
```

**Ограничения:**
- Нельзя пригласить пользователя с ролью выше своей.
- Одному email может соответствовать только один аккаунт в системе.
- Истёкшее приглашение можно отозвать и выслать повторно.

### 5.3 Карточка пользователя

При клике на сотрудника открывается боковая панель:
- ФИО, email, аватар
- Назначенные роли по точкам (можно редактировать)
- Последний вход в систему
- Активные сессии
- Кнопки: «Изменить роль», «Заблокировать», «Удалить»

## 6. Аутентификация

| Параметр | Значение |
|----------|---------|
| Метод | Email + пароль, опционально Google OAuth 2.0 |
| 2FA | TOTP (Google Authenticator / Authy), опционально; обязателен для `owner` и `fin_controller` |
| Сессия | JWT access token (15 мин) + refresh token (30 дней) |
| Хранение паролей | bcrypt, cost factor ≥ 12 |
| Политика паролей | Минимум 8 символов, 1 цифра, 1 буква верхнего регистра |
| Блокировка | 5 неудачных попыток → CAPTCHA; 10 неудачных → блокировка аккаунта на 30 мин |

## 7. Изоляция данных

### 7.1 Правила видимости

- Пользователь видит **только данные тех точек**, к которым он привязан.
- `ops_director` и `owner` на уровне организации видят **все точки**.
- Нет механизма «гостевого» доступа к данным другой организации.

### 7.2 Row-level security (RLS)

На уровне БД (PostgreSQL) для ключевых таблиц реализован RLS:

```sql
-- Пример: политика для таблицы orders
CREATE POLICY orders_venue_isolation ON orders
    USING (venue_id = ANY(current_setting('app.user_venue_ids')::uuid[]));
```

Параметр `app.user_venue_ids` устанавливается при открытии соединения из API-слоя на основе JWT-токена.

## 8. Audit Trail (журнал событий)

Все действия, изменяющие данные, пишутся в таблицу `audit_log`:

```sql
CREATE TABLE audit_log (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    org_id      UUID NOT NULL,
    venue_id    UUID,
    user_id     UUID NOT NULL REFERENCES users(id),
    action      TEXT NOT NULL,   -- 'create','update','delete','approve','login'
    entity_type TEXT NOT NULL,   -- 'order','supplier','user','recipe'
    entity_id   UUID,
    payload     JSONB,           -- diff: {before: {...}, after: {...}}
    ip_address  INET,
    user_agent  TEXT,
    created_at  TIMESTAMPTZ DEFAULT now()
);
```

**Ретенция:** журнал хранится 2 года. Удаление записей журнала запрещено даже для `owner`.

## 10. Критерии приёмки

- [ ] Пользователь без нужной роли не может видеть раздел UI и получает 403 при прямом API-запросе.
- [ ] При смене роли изменения вступают в силу без перезахода (на следующем запросе к API).
- [ ] Приглашение истекает через 72 ч; после истечения ссылка возвращает понятное сообщение.
- [ ] Нельзя назначить пользователю роль, выше собственной.
- [ ] Все изменения данных (create/update/delete) фиксируются в audit_log с diff.
- [ ] Owner всегда может посмотреть полный список активных сессий и завершить любую.
- [ ] 2FA обязателен для `owner` и `fin_controller`; при входе без 2FA — принудительный redirect на настройку.
- [ ] RLS проверяется интеграционным тестом: пользователь A не может прочитать данные точки, к которой не привязан, даже если знает venue_id.
