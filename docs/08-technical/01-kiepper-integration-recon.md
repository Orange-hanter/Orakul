# Kiepper Integration Recon

> Этот документ — живой. Заполняется по мере получения ответов от вендора, анализа API и пробных запросов. Каждый открытый вопрос помечен `[ ]`; закрытый — `[x]` с ответом.

## 0. Цель

Определить **точный метод доступа к данным Kiepper** и задокументировать контракт интеграции до начала написания ETL-кода. Без этого документа реализация коннектора — игра в угадайку.

**Критический вопрос:** _как именно Kiepper отдаёт данные_ — этот ответ определяет всю архитектуру ETL и частоту обновления прогнозов.

## 1. Методы доступа — анализ рынка

На основе анализа российских систем автоматизации общепита (iiko, r_keeper, Poster, 1С:Общепит) выявлены 4 паттерна, которые встречаются у подобных Kiepper систем. Нужно определить, какой из них применим.

### Паттерн A: REST API (предпочтительный)
Большинство современных систем имеют документированный REST API.

**Признаки:**
- В настройках Kiepper есть раздел «API» или «Интеграции»
- Вендор выдаёт токен или ключ разработчика
- Есть документация вида `api.kiepper.com/docs`

**Плюсы:** инкрементальный pull, почти реальное время, стандартный JSON.  
**Что проверить:**
```
[ ] Базовый URL API
[ ] Формат аутентификации (Bearer token / API key / OAuth2)
[ ] Rate limits (запросов в минуту/час)
[ ] Поддержка инкрементального pull (параметр ?from= / ?updated_since= / cursor)
[ ] Webhook-поддержка (push от Kiepper при изменении)
```

### Паттерн B: Прямой доступ к БД (PostgreSQL / MSSQL)
Встречается у on-premise инсталляций (iiko on-premise, r_keeper).

**Признаки:**
- Kiepper установлен локально на сервере клиента
- IT-администратор имеет доступ к БД-серверу
- Вендор предоставляет схему БД или read-only пользователя

**Плюсы:** полный доступ к данным, любая частота.  
**Минусы:** зависимость от внутренней схемы вендора (может меняться без предупреждения, риск R-05).  
**Что проверить:**
```
[ ] СУБД (PostgreSQL / MSSQL / MySQL)
[ ] Есть ли read-replica или read-only пользователь?
[ ] Задокументирована ли схема вендором?
[ ] Версионирование схемы — уведомляют ли об изменениях?
```

### Паттерн C: Файловый экспорт (CSV / XML / Excel)
Самый простой, но и самый ограниченный.

**Признаки:**
- Kiepper умеет «выгрузить отчёт» в XLS/CSV
- Нет API-документации
- Экспорт запускается вручную или по расписанию в интерфейсе

**Плюсы:** не нужна интеграция, достаточно парсера.  
**Минусы:** частота — раз в сутки максимум, ручная работа, нет реального времени.  
**Что проверить:**
```
[ ] Доступна ли автоматическая/scheduled выгрузка (SFTP, email, папка)?
[ ] Какой формат: CSV, XLS, XML?
[ ] Стабильна ли структура файла между версиями Kiepper?
[ ] Есть ли выгрузка нужных нам сущностей (список в §3)?
```

### Паттерн D: Webhook / Event stream
Встречается у современных cloud-систем.

**Признаки:**
- Kiepper cloud-версия
- В настройках можно указать URL для получения событий
- Push при каждой продаже / поступлении

**Плюсы:** реальное время.  
**Минусы:** нужен публичный endpoint, усложняет инфраструктуру.

## 4. Анализ рынка — паттерны схем данных

На основе публичных API аналогичных систем (iiko API v2, Poster API, r_keeper JSON API) зафиксированы типичные паттерны, на которые можно ориентироваться при разведке Kiepper:

### 4.1 Типичные ID-форматы
- UUID v4 — iiko (предпочтительно: стабильны)
- Целочисленные инкременты — r_keeper (риск коллизий при multi-tenant)
- Строковые коды — Poster (могут меняться при переименовании позиции)

**Наш приоритет:** UUID или числовой surrogate key, который **не меняется** при переименовании позиции.

### 4.2 Типичные проблемы с единицами измерения
В большинстве систем встречаются:
- Хранение нормы в граммах, но отображение в «кг» в UI → нужна конвертация
- Смешанные единицы в одной таблице (г и мл и шт без явного типа)
- Ингредиент с нормой «1 порция» без расшифровки — мёртвая рецептура

**Что делаем:** на уровне `staging` нормализуем всё к базовой единице (грамм для весовых, мл для жидких, шт для штучных). Конвертационная таблица — в конфиге коннектора.

### 4.3 Типичные проблемы с историей продаж
- Чек без детализации по блюдам (только сумма) — встречается в старых версиях POS
- Удалённые чеки остаются в БД с флагом `is_deleted=true` — нужна явная фильтрация
- Перенос данных при смене ПО оставляет «дыры» в истории

### 4.4 Типичные проблемы с рецептурами
- Рецептура создаётся 1 раз при открытии заведения и больше не обновляется — данные устарели
- Существуют «технологические карты» (полуфабрикаты) с вложенностью: блюдо → полуфабрикат → ингредиент. Нужно разворачивать рекурсивно.
- Модификаторы/опции блюда (убрать лук, добавить сыр) хранятся отдельно и тоже имеют нормы закладки — часто забывают учесть.

## 5. Стратегия коннектора (обновляется после разведки)

> Заполняется после определения паттерна доступа.

### Если Паттерн A (REST API):

```python
# Псевдокод инкрементального pull
class KiepperConnector:
    def pull_incremental(self, entity: str, since: datetime) -> list[dict]:
        """
        GET /api/v1/{entity}?updated_since={since}&limit=500&offset=0
        Пагинация до пустого ответа.
        Сохраняем raw JSONB в raw.{entity} с import_ts.
        """
        ...

    def pull_full(self, entity: str) -> list[dict]:
        """
        Только для справочников (SKU, рецептуры, меню).
        Запускается 1 раз при старте + ежедневно в 02:00.
        """
        ...
```

**Расписание:**
| Сущность | Тип pull | Расписание |
|----------|----------|-----------|
| SKU / Меню | full_refresh | 1 раз в сутки, 02:00 |
| Рецептуры | full_refresh | 1 раз в сутки, 02:00 |
| Продажи | incremental | каждые 15 мин |
| Поставки | incremental | каждые 30 мин |
| Остатки | incremental | каждые 30 мин или по событию |
| Списания | incremental | каждые 30 мин |

### Если Паттерн B (прямая БД):

```sql
-- Инкрементальный pull продаж
SELECT * FROM kiepper.sales_documents
WHERE updated_at > :last_run_ts
  AND status = 'closed'
ORDER BY updated_at;
```

**Риск:** при обновлении Kiepper схема может измениться без уведомления (R-05). Митигация: слой `raw` хранит дамп, `staging` трансформация легко обновляется.

### Если Паттерн C (CSV-экспорт):

- Частота: раз в сутки (ограничение паттерна)
- Парсер: `pandas.read_csv` / `openpyxl` для Excel
- Хранение: тот же `raw` слой, но поле `source = 'csv_export'`
- **Важно:** закладывать в прогноз задержку — данные вчерашнего дня будут только утром

## 6. Тест-план первого подключения (Day 1)

После получения доступа выполнить последовательно:

```
Шаг 1: Аутентификация
  → Получить токен / подключиться к БД / скачать тестовый файл
  → Результат: успешный ответ без ошибок 401/403

Шаг 2: Выгрузка справочника SKU
  → Получить полный список активных ингредиентов
  → Проверить: есть ли поля id, name, unit, is_active
  → Посчитать: сколько всего SKU, сколько без unit

Шаг 3: Выгрузка рецептур
  → Получить рецептуры для первых 20 блюд
  → Проверить: ingredient_id есть в справочнике SKU
  → Посчитать: % блюд с полными рецептурами

Шаг 4: Выгрузка продаж за последние 7 дней
  → Проверить: детализация до блюд есть в каждом чеке
  → Посчитать: кол-во чеков, топ-10 блюд по продажам
  → Совпадает ли с ощущениями клиента? (sanity check)

Шаг 5: Выгрузка остатков (последняя инвентаризация)
  → Проверить наличие данных
  → Сравнить со знанием клиента: «молока реально ~15 кг»

Шаг 6: Запустить Data Audit L1-проверки
  → По критериям из [Data Audit Checklist](../04-implementation/04-data-audit-checklist-kiepper.md)
  → Зафиксировать результаты в §7 этого документа
```
