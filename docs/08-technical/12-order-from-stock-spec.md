# Спецификация модуля «Заказ от склада»

## 0. TL;DR

Новая точка входа в формирование заявки: **склад → low-stock товары → выбор поставщика per item → отправка через email/Telegram/Viber диплинк**. Существующий [OrderWizard](07-order-request-spec.md) (вход от поставщика) сохраняется как fallback для разовых заказов без привязки к остаткам.

Переиспользует существующий движок рекомендаций ([app/client/src/utils/recommendations.js](../../app/client/src/utils/recommendations.js)): `currentStockForProduct`, `cheapestSupplierForProduct`, `recommendForProduct`, `forecastProductConsumption`.

## 1. Назначение

Текущий флоу «поставщик → позиции» неудобен для **типового кейса** закупки: «у меня заканчиваются продукты, надо это пополнить». Закупщик в реальности отталкивается **от остатков склада**, а не от каталога поставщика. Новая точка входа разворачивает флоу: сначала юзер видит, что заканчивается, потом система предлагает самого дешёвого поставщика per item, потом сообщение уходит поставщику через привычный канал (почта/мессенджер).

## 2. Точки входа

| Точка входа | Триггер |
|---|---|
| Вкладка «Склад» → чекбоксы у low/out товаров → sticky-bar «Заказать (N) →» | Bulk-заказ нескольких товаров |
| Вкладка «Склад» → кнопка «Заказать» на одной карточке | Single-item shortcut |
| Бейдж «Заканчивается» / «Закончился» — кликабельный | То же, что single-item shortcut |
| Вкладка «Заявки» → «Создать вручную» | Открывает старый [OrderWizard](07-order-request-spec.md) для заказов без привязки к остаткам |

## 3. Изменения модели данных

### 3.1 Добавить поля

**`product`** — порог ручного перезаказа:

| Поле | Тип | Описание |
|---|---|---|
| `reorderPoint` | `number \| null` | Минимальный остаток в единицах товара. Если `null` — для определения low-stock используется только forecast. |

**`supplier`** — контактные каналы (свободнотекстовый `contact` остаётся для прочей информации):

| Поле | Тип | Описание |
|---|---|---|
| `email` | `string \| null` | Адрес для `mailto:` диплинка. |
| `telegramUsername` | `string \| null` | Без `@`. Также допускается `+375...` для приватного чата. |
| `viberPhone` | `string \| null` | E.164 (`+375...`). |

**`order`** — трекинг отправки:

| Поле | Тип | Описание |
|---|---|---|
| `sentVia` | `'email' \| 'telegram' \| 'viber' \| null` | Какой канал юзер выбрал. |
| `sentAt` | `number \| null` | Timestamp клика «Отправить» (намерение, не доставка). |
| `messageText` | `string \| null` | Снэпшот сгенерированного текста — для аудита и повторной отправки. |

### 3.2 Семантика статусов

| Статус | Условие |
|---|---|
| `draft` | Создан, но не отправлен. |
| `submitted` | Юзер кликнул «Отправить через X» (диплинк открыт). Заполнены `sentVia` + `sentAt`. |
| `received` | Юзер вручную отметил приёмку. Заполнено `receivedAt`. |

### 3.3 Миграции

Не требуются — все новые поля опциональны. Старые `order` без `sentVia` остаются совместимы.

## 4. Сигнал «заканчивается»

Новый чистый хелпер [app/client/src/utils/stockStatus.js](../../app/client/src/utils/stockStatus.js):

```js
function stockStatusForProduct(records, productId) {
  const product = records.find(r => r.type === 'product' && r.id === productId);
  const stock   = currentStockForProduct(records, productId);
  const rec     = recommendForProduct(records, productId);

  let status;
  if (stock === 0)                                              status = 'out';
  else if (product.reorderPoint && stock < product.reorderPoint) status = 'low-manual';
  else if (rec !== null)                                         status = 'low-forecast';
  else                                                           status = 'ok';

  return {
    status,
    stock,
    reorderPoint:     product.reorderPoint,
    suggestedQty:     rec?.suggestedQty ?? null,
    cheapestSupplier: rec?.factors.cheapestSupplier ?? null,
    factors:          rec?.factors ?? null,
  };
}
```

**Приоритеты:**
1. `out` (stock = 0) — всегда красный, безусловно.
2. `low-manual` — ручной порог. Прозрачно для юзера и работает на товарах без рецепта/продаж (упаковка, расходники).
3. `low-forecast` — forecast > stock. Учитывает рост спроса.

`factors` нужен для tooltip-объяснения бейджа («3 кг на 2 дня + 1 день буфер, остаток 2 кг»).

## 5. UI

### 5.1 Вкладка «Склад»

| Статус | Бейдж | Поведение |
|---|---|---|
| `out` | 🔴 «Закончился» | Чекбокс предвыбран по умолчанию при открытии вкладки |
| `low-manual` | 🟠 «Ниже порога» + tooltip с reorderPoint |  |
| `low-forecast` | 🟠 «Заканчивается» + tooltip с прогнозом |  |
| `ok` | (без бейджа) | Чекбокс есть, но скрыт за hover/чекбокс-режимом |

**Форма редактирования продукта** — новое поле «Порог заказа (опц.), ед.».

### 5.2 Модал `OrderFromStock` — Шаг 1 (Корзина)

Per-line настройка для каждого выбранного товара:

```
┌─ Сахар-песок · остаток 1 кг ─────────────────────────────┐
│ Количество: [ 30 ] кг                                    │
│ ℹ️ Рекомендую 30: forecast 25 + min-партия 50→30          │
│                                                           │
│ Поставщик:                                                │
│   ● ИП Козлов        1.20 BYN/кг   🟢 самый дешёвый      │
│   ○ АгроПоставка     1.40 BYN/кг                         │
│                                                           │
│ Цена позиции: 36.00 BYN                                  │
└──────────────────────────────────────────────────────────┘
```

**Логика выбора поставщика:**
- Источник — все `supplier_item` с `productId === product.id`, отфильтрованные по `supplier.status !== 'paused'`.
- По умолчанию выбран самый дешёвый (через `cheapestSupplierForProduct`).
- Если у товара **нет ни одного supplier_item с этим productId** — показываем все активные поставщики + поле «название позиции» для свободного ввода (item-snapshot, без записи в каталог поставщика).

**Предупреждение** если `qty < minQty` поставщика: «Минимальная партия 50 кг, заказ может быть отклонён».

**Footer — авто-группировка**:


## 8. Бэкенд

**Минимум:** изменений в эндпоинтах не требуется. POST `/api/records` уже примет новые поля через `{ ...req.body }` без allowlist'а ([app/server/records.js:52](../../app/server/records.js#L52)).

**Опционально (фаза 4):** `GET /api/orders/:id/message` — серверная генерация текста по тому же шаблону. Полезно если шаблон станет конфигурируемым per-venue или per-supplier.

## 9. Каналы — особенности

| Канал | Диплинк | Лимит | Fallback |
|---|---|---|---|
| **Email** | `mailto:${email}?subject=${enc}&body=${enc}` | ~2000 симв. URL в Outlook/Gmail-Web | Если текст > 1800 симв. — открыть пустое `mailto:` + скопировать тело в clipboard, toast «Текст скопирован» |
| **Telegram** | `https://t.me/${username}?text=${enc}` | ~4096 симв. (telegram message limit) | Если у поставщика только номер: `https://t.me/${phone}` без `text` + clipboard |
| **Viber** | `viber://chat?number=${phone}` | Не поддерживает `text=` параметр | Всегда копировать текст в clipboard перед открытием. На десктопе без Viber — graceful fail с toast |

## 10. Фазы реализации

| Фаза | Объём | Критерий готовности |
|---|---|---|
| **1 — Сигнал** | `reorderPoint` в product UI; `stockStatus.js`; бейджи на карточках склада + tooltip с factors | На вкладке «Склад» low/out товары визуально отличимы; редактирование порога работает |
| **2 — Корзина** | Чекбоксы + sticky-bar; `OrderFromStock` шаг 1; авто-группировка; создаёт `order(status: 'draft')` без отправки | Можно выбрать N товаров и получить N черновиков заявок с автогруппировкой |
| **3 — Отправка** | Поля контактов в `supplier` form; шаг 2 модала; `orderMessage.js`; диплинки email/tg/viber; поля `sentVia/sentAt/messageText` в order | Один клик создаёт submitted-order и открывает диплинк |
| **4 — Полировка** | «Отправить все»; clipboard-fallback для Viber и длинных писем; переотправка из истории | UX-шлифовка |

## 11. Открытые риски

- **Viber на десктопе:** диплинк работает только при установленном Viber Desktop. UX-стратегия: всегда копировать текст в clipboard перед открытием → юзер сможет вставить вручную если диплинк не сработал.
- **`mailto:` лимит длины:** ~2000 символов — заказ на 50+ позиций может не влезть. Решается в фазе 4 через clipboard-fallback.
- **Telegram username vs. phone:** поле `telegramUsername` принимает оба формата (`username` или `+375...`). Валидация — только базовая (regex), без проверки существования в Telegram.
- **Снэпшот цен:** в момент клика «Отправить» цены в `order.items[]` фиксируются из текущих `supplier_item.price`. Если каталог обновят позже — исторический order остаётся корректным.
- **Несколько venue:** контакты поставщика глобальные (supplier не venue-scoped), но в тексте письма подставляется `venue.name` текущего контекста — поставщик видит, для какой точки заказ.

## 12. Не входит в scope (отложено)

- Авто-отправка с сервера (SMTP / Telegram-бот шлёт сам). Пока — только диплинки.
- PDF/XLSX экспорт заявки (описан в [Order Request Spec §10](07-order-request-spec.md)).
- Согласование manager → ops_director (требует RBAC, описано в [RBAC Spec](08-rbac-multiuser-spec.md)).
- Подтверждение доставки от поставщика по диплинку (нет петли обратной связи — статус `received` юзер ставит вручную).
- Конфигурируемый шаблон сообщения (per-venue, per-supplier, мультиязычность).
