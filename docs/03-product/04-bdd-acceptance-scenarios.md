# BDD Acceptance Scenarios

> Этот документ — **формальные критерии приёмки** в нотации Given / When / Then. Бизнес-заказчик может проверить эти сценарии без программиста — через интерфейс и реальные данные.

Соответствие User Stories: каждая US-XX покрыта ≥3 сценариями (golden path + 2 edge cases).

---

## 0. Принципы BDD в Orakul

1. **Сценарий описывает поведение, а не реализацию.** Не «нажал на кнопку», а «увидел рекомендацию с обоснованием».
2. **Каждый сценарий — независим.** Не имеет hidden state из других сценариев.
3. **Контекст (Given) описывает ВСЕ предусловия.** Если сценарий «работает только когда есть рецептура» — это часть Given.
4. **Then проверяет наблюдаемый результат.** Не «модель пересчитала» (внутреннее), а «пользователь увидел X» (наблюдаемое).
5. **Negative scenarios обязательны.** Не только golden path.

## 1. US-01 — Алёрты по остаткам и срокам годности

### Сценарий 1.1 (golden) — Алёрт за 24 часа до риска
```
Given: точка X имеет SKU «куриная грудка» с остатком 8 кг
And:   средняя продажа SKU за последние 4 недели = 5 кг/день
And:   следующая поставка ожидается через 48 часов
And:   управляющий точки X имеет роль «manager» и активную сессию
When:  Orakul пересчитывает прогноз остатков (каждые 15 минут)
Then:  управляющий получает push-уведомление в течение 5 минут с заголовком
       «Закончится за 24 часа: куриная грудка»
And:   уведомление содержит: текущий остаток (8 кг), прогноз использования (5 кг/день),
       рекомендуемое действие (увеличить заказ на N кг или ускорить поставку),
       источник данных (последние продажи)
And:   уведомление доступно в журнале не менее 90 дней
```

### Сценарий 1.2 (edge) — Управляющий отмечает «отложено»
```
Given: управляющий получил алёрт по куриной грудке (см. 1.1)
When:  управляющий нажимает «отложено» и комментирует «партия в пути, прибудет за 6 ч»
Then:  алёрт меняет статус на «отложено» с timestamp и комментарием
And:   повторный алёрт по той же позиции не приходит в течение 6 ч
And:   через 6 ч система пересчитывает; если остаток всё ещё критический — повторный алёрт
And:   действие фиксируется в audit-trail (US-08)
```

### Сценарий 1.3 (negative) — Нет данных по продажам, алёрт не показывается
```
Given: SKU «новый_ингредиент» добавлен 2 дня назад
And:   нет истории продаж за период ≥7 дней
When:  Orakul пересчитывает остатки
Then:  алёрт по «новый_ингредиент» НЕ формируется
And:   в журнале фиксируется запись «модель воздержалась от рекомендации: недостаточно данных (минимум 7 дней)»
And:   у пользователя в админке есть индикатор «N SKU без прогноза» с пояснением
```

## 2. US-02 — Сравнение факт vs норма (списания)

### Сценарий 2.1 (golden) — Отклонение свыше порога подсвечивается
```
Given: блюдо «бургер классик» имеет норму закладки 180 г фарша
And:   за смену продано 50 порций
And:   списано фарша 11 кг (норма = 9 кг, отклонение +22%)
And:   порог подсветки настроен на ≥15%
When:  смена закрывается
Then:  отчёт «факт vs норма» содержит строку «бургер классик: +22% к норме» с красной маркировкой
And:   к строке привязаны: смена, ответственный (по графику), фактическое списание, норма
And:   управляющий или шеф может добавить причину отклонения (свободный текст или из списка: «новый сотрудник», «изменение поставщика», «технологическая проба»)
```

### Сценарий 2.2 (edge) — Отклонение в пределах порога
```
Given: те же условия, отклонение +12% (порог 15%)
When:  смена закрывается
Then:  строка отражена в отчёте без подсветки (нейтрально)
And:   аномалия НЕ создаётся, push не отправляется
```

### Сценарий 2.3 (audit) — Экспорт для месячного аудита
```
Given: финконтролёр имеет роль «finance»
And:   за месяц зафиксировано 47 отклонений >15%
When:  финконтролёр запрашивает экспорт «отчёт по аномалиям за месяц»
Then:  получает CSV или PDF со столбцами: дата, смена, точка, блюдо, факт, норма, отклонение, причина (если указана), ответственный, статус
And:   получает аудит-метку: «выгружено [user] [timestamp]» в журнале
```

## 3. US-03 — Рекомендации по заказам

### Сценарий 3.1 (golden) — Формирование проекта заказа
```
Given: точка X, текущая дата = вторник
And:   прогноз спроса на ближайшие 5 дней рассчитан
And:   остатки на складе известны
And:   срок поставки от поставщика = 24 ч
When:  закупщик открывает экран «рекомендуемые заказы»
Then:  видит таблицу: SKU, текущий остаток, прогноз потребления, рекомендуемое количество, обоснование
And:   обоснование для каждой строки включает: прогноз спроса, минимальную партию поставщика, срок годности, активные акции (если есть)
And:   может корректировать количество вручную; правка сохраняется с причиной (свободный текст)
```

### Сценарий 3.2 (golden) — Утверждение и передача в Kiepper
```
Given: закупщик утвердил проект заказа из сценария 3.1
When:  нажимает «передать в Kiepper»
Then:  в Kiepper создаётся запись «заказ поставщику» в статусе «черновик»
And:   в Orakul состояние заказа = «передан в Kiepper», дата и пользователь зафиксированы
And:   в audit-trail фиксируется событие RECOMMENDATION_OUTCOME с типом «accepted_to_kiepper»
```

### Сценарий 3.3 (negative) — Низкое качество данных, рекомендация не выдается
```
Given: точка X имеет 35% SKU без рецептур (см. Data Audit, R-04)
When:  закупщик открывает экран
Then:  для SKU без рецептур показано «недостаточно данных для прогноза» вместо рекомендации
And:   общий бейдж «качество данных: 65%» виден в шапке экрана
And:   ссылка на инструкцию «как улучшить»
```

## 4. US-04 — Сравнение поставщиков

### Сценарий 4.1 (golden) — Просмотр сравнительной таблицы
```
Given: SKU «оливковое масло 5 л» имеет 4 поставщика
And:   у каждого есть актуальная цена, минимальная партия, срок доставки, рейтинг надёжности
When:  закупщик открывает «сравнить поставщиков» для SKU
Then:  видит таблицу с 4 строками; столбцы: поставщик, цена/единицу, мин. партия, срок доставки, рейтинг, тренд цены за 90 дней
And:   может зафиксировать выбранного поставщика и добавить комментарий
And:   фиксация фиксируется в audit-trail
```

### Сценарий 4.2 (edge) — Свежесть прайса
```
Given: один из 4 поставщиков прислал прайс 75 дней назад
When:  таблица отображается
Then:  у этого поставщика бейдж «прайс устарел» (>30 дней), цена показана с предупреждением
And:   фильтр «только актуальные цены» позволяет скрыть устаревшие
```

### Сценарий 4.3 (audit) — История изменений
```
Given: закупщик хочет понять, как менялась цена SKU у поставщика A
When:  кликает «история цен»
Then:  видит график за 12 мес с датами каждого изменения
And:   видит источник каждого значения (прайс, ручной ввод, накладная)
```

## 5. US-05 — Маржинальность блюд в реальном времени

### Сценарий 5.1 (golden) — Пересчёт при изменении закупочной цены
```
Given: блюдо «паста карбонара» имеет food cost = 32%, маржу = 68%
And:   рецептура содержит ингредиент «гуанчале» (норма 50 г, цена 1 800 ₽/кг)
When:  закупочная цена «гуанчале» меняется на 2 100 ₽/кг
Then:  food cost блюда автоматически пересчитывается в течение 5 мин
And:   шеф-повар получает алёрт «маржа изменилась: «карбонара» 68% → 65%»
And:   алёрт содержит ссылку на drill-down до ингредиентов
```

### Сценарий 5.2 (edge) — Изменение нормы закладки
```
Given: шеф-повар меняет норму «гуанчале» с 50 г на 45 г
When:  сохраняет рецептуру в Kiepper
Then:  food cost блюда пересчитывается в течение 5 мин
And:   видно сравнение «до/после»: 32% → 30%
And:   изменение залогировано (кто, когда, что было, что стало)
```

### Сценарий 5.3 (export) — Экспорт для согласования
```
Given: шеф-повар выбрал блюдо
When:  нажимает «экспорт»
Then:  получает PDF с: фото блюда, рецептурой, нормами, ценами на дату, food cost, маржой, трендом за 3 мес
And:   в имени файла дата и пользователь
```

## 6. US-06 — Сценарное моделирование меню

### Сценарий 6.1 (golden) — Сценарий «снизить цену на N%»
```
Given: собственник открывает «сценарное моделирование» для блюда «лосось гриль»
And:   текущая цена 1 200 ₽, продажи 30/день, маржа 55%
When:  собственник в слайдере выставляет «−10% к цене»
Then:  получает прогноз: ожидаемые продажи (например 36/день, +20%), новая маржа (47%), влияние на расход ингредиентов (+150 г лосося/день), изменение прибыли (₽/день)
And:   видит блок «допущения и риски» (эластичность спроса 1.5, не учтена сезонность, текущая цена лосося)
```

### Сценарий 6.2 (golden) — Сравнение сценариев
```
Given: собственник создал 3 сценария для одного блюда
When:  открывает «сравнить»
Then:  видит таблицу сценариев в одних осях: продажи, маржа, расход, прибыль
And:   может пометить «утверждено» для одного из сценариев
And:   может «архивировать» отвергнутые
```

### Сценарий 6.3 (edge) — Слишком агрессивный сценарий
```
Given: собственник выставляет «−40% к цене» (за пределами модели эластичности)
When:  применяет сценарий
Then:  система предупреждает «сценарий выходит за пределы исторических данных, прогноз менее надёжен»
And:   доверительный интервал прогноза расширяется визуально
```

## 7. US-07 — Сетевой дашборд

### Сценарий 7.1 (golden) — Сравнение точек
```
Given: опердир открывает сетевой дашборд
And:   подключено 12 точек
When:  применяет фильтр «за последний месяц», группировка по «формат»
Then:  видит таблицу с метриками BIZ-01..04 по каждой точке + цветовой код
And:   сортировка по любой колонке
And:   drill-down до уровня точки одним кликом
```

### Сценарий 7.2 (edge) — Точка без данных
```
Given: точка-12 открыта 2 дня назад, истории недостаточно
When:  опердир смотрит дашборд
Then:  точка-12 показана с бейджем «новая, метрики формируются» вместо цветового кода
```

### Сценарий 7.3 (rollout) — Еженедельная рассылка
```
Given: подписка на еженедельный срез включена
When:  понедельник 09:00
Then:  опердир получает email с PDF-снимком дашборда
And:   в письме — топ-3 «победы недели» и топ-3 «риски»
```

## 8. US-08 — Объяснимость рекомендации

### Сценарий 8.1 (golden) — Развёрнутое обоснование
```
Given: пользователь видит рекомендацию по заказу «увеличить молоко 3.2% на 50 кг»
When:  раскрывает «обоснование»
Then:  видит: прогноз спроса (24 кг/день × 4 дня = 96 кг), текущий остаток (40 кг), рекомендуемая поставка (50 кг с учётом мин. партии 25 кг), используемые факторы (история продаж 90 дней, активная акция «кофе+круассан»)
```

### Сценарий 8.2 (golden) — Обратная связь
```
Given: пользователь раскрыл обоснование
When:  нажимает «👎» и комментирует «акция уже закончилась 2 дня назад»
Then:  feedback фиксируется в журнале
And:   для этой акции/SKU модель пересчитывается с поправкой
And:   пользователь видит сообщение «спасибо, учли при следующем расчёте»
```

### Сценарий 8.3 (negative) — Рекомендация не показывается без обоснования
```
Given: модель сгенерировала рекомендацию, но обоснование не сформировалось (баг шаблонизатора)
When:  Orakul пытается показать рекомендацию
Then:  рекомендация НЕ показывается пользователю (принцип PRD §3 «Always explain»)
And:   событие фиксируется как инцидент уровня P2
And:   в журнале состояние = «expired due to missing rationale»
```

## 9. Кросс-функциональные сценарии

### Сценарий K.1 — Свежесть данных Kiepper нарушена
```
Given: интеграция с Kiepper не получила обновление >2 ч (KPI OPS-05)
When:  пользователь открывает экран рекомендаций
Then:  баннер «данные актуальны на [timestamp], идёт восстановление»
And:   новые рекомендации не формируются до восстановления
And:   уже сформированные рекомендации помечены «возможно устарели»
And:   PMO + TL получают P1 алёрт
```

### Сценарий K.2 — Изменение схемы Kiepper
```
Given: Kiepper выпустил версию с изменённым полем рецептуры
When:  Orakul обнаруживает несовпадение схемы
Then:  включается процедура «диагностика и блокировка» (PRD §8)
And:   рекомендации, зависящие от изменённого поля, не показываются
And:   PMO получает алёрт; запускается митигация R-05
```

### Сценарий K.3 — RBAC: попытка доступа сверх прав
```
Given: управляющий точки A пытается открыть метрики точки B
When:  отправляет запрос
Then:  получает 403 / экран «нет доступа»
And:   событие фиксируется в audit-trail безопасности
And:   при N таких попыток за час — алёрт безопаснику
```
