# Product Requirements Document (PRD)

## 1. Назначение и аудитория

PRD — мост между **бизнес-требованиями** (BRD) и **функциональными требованиями к реализации**. Документ:
- описывает, **что** продукт делает с точки зрения пользователя и какие задачи решает;
- задаёт **продуктовые контуры** (модули, состояния, события, права доступа);
- фиксирует **продуктовые принципы и ограничения**;
- ссылается на User Stories (детали сценариев), BDD (критерии приёмки), Roadmap (сроки).

PRD читают: продакт, дизайнер, разработка, QA, документация, CSM. Бизнес-стейкхолдеры читают только §2–4 и §11.

## 2. Контекст продукта (1 страница для бизнеса)

### 2.1 Что решаем
Управление общепитом сегодня — реактивное: списания и стоп-листы вскрываются постфактум, закупки делаются «на глаз», маржинальность пересчитывается раз в месяц. Цена этой реактивности — 5–9% оборота, теряемых на потерях и неоптимальных закупках.

### 2.2 Что предлагаем
Orakul — **цифровой советник** на базе ИИ, который читает данные Kiepper и формирует **рекомендации к действию** для управляющего, закупщика, шеф-повара и финконтролёра. Каждая рекомендация **объяснима** и сохраняет аудит-след.

### 2.3 Что НЕ предлагаем
Не подменяем Kiepper, не делаем POS, не управляем персоналом, не делаем CRM/лояльность, не подключаем IoT. См. [BRD §3](../02-business/01-business-requirements-document.md).

## 3. Принципы продукта

| Принцип | Что значит при принятии решений |
|---------|----------------------------------|
| **Recommend, don't act** | Orakul не действует за пользователя без явного подтверждения. Авто-действия включаются только под отдельным CR с обоснованием. |
| **Always explain** | Любая рекомендация имеет блок «Обоснование» с исходными факторами. Без обоснования — не показывается. |
| **One job per screen** | Каждый экран отвечает на один вопрос: «что делать сейчас», «что заказать», «где перерасход». Многоцелевые экраны запрещены. |
| **Time-of-day UX** | Сценарии разные утром, в обед и вечером. Главный экран — динамический. |
| **Default safe** | По умолчанию пользователь всегда может откатить, отметить «принято/отложено» или дать обратную связь. Никаких «согласовать без подтверждения». |
| **No black box** | Если модель не уверена (вероятность <порога) — рекомендация не показывается, а в журнале фиксируется «модель воздержалась». |
| **Latency budget** | Рекомендация рассчитывается ≤15 минут от события или запроса (BRD §7). Превышение — алёрт, не отображение «крутится колесо». |

## 6. Состояния и события рекомендации

### 6.1 Жизненный цикл рекомендации

```
[Сгенерирована] → [Показана пользователю] → [Действие пользователя] → [Закрыта]
                                                ↓
                          ┌─────────────────────┼─────────────────────┐
                          ↓                     ↓                     ↓
                    Принято         Скорректировано         Отклонено
                  (исполнено         (с правкой и           (с обоснованием)
                   как есть)          обоснованием)
```

### 6.2 Возможные состояния
| Состояние | Описание |
|-----------|----------|
| `draft` | Сгенерирована, но не показана (модель не уверена / в очереди) |
| `pending` | Показана, ожидает действия |
| `accepted` | Принята как есть |
| `adjusted` | Принята с корректировкой пользователя |
| `rejected` | Отклонена с обоснованием |
| `ignored` | Не отреагировали в течение SLA-окна |
| `expired` | Утратила актуальность (изменились данные) |

### 6.3 События для аудит-трейла
- `RECOMMENDATION_GENERATED` (с входными факторами)
- `RECOMMENDATION_SHOWN` (кому, когда)
- `USER_ACTION` (тип, комментарий)
- `RECOMMENDATION_OUTCOME` (что произошло после, через 24/72 часа)

> Все события сохраняются ≥3 года для целей аудита (BRD §7, [Data Governance](../05-governance/03-data-governance.md)).

## 7. Роли и права доступа (RBAC)

| Роль | Создание/правка рецептур | Утверждение заказов | Просмотр всех точек | Просмотр финансовых KPI | Экспорт |
|------|---------------------------|----------------------|----------------------|--------------------------|---------|
| Собственник | ❌ | ❌ | ✅ | ✅ | ✅ |
| Опердир | ❌ | ✅ (свыше лимита) | ✅ | ✅ | ✅ |
| Финконтролёр | ❌ | ❌ | ✅ | ✅ | ✅ |
| Управляющий точкой | ❌ | ✅ (свой лимит) | ❌ (только своя) | ✅ (своя точка) | ⚠ (по разрешению) |
| Закупщик | ❌ | ⚠ (черновик) | ✅ (закупочный пул) | ✅ (только закупки) | ✅ |
| Шеф-повар | ✅ | ❌ | ❌ | ✅ (food cost) | ✅ |
| Аудитор / read-only | ❌ | ❌ | ✅ | ✅ | ✅ |

> Принципы: **least privilege**, **separation of duties** (создание рецепта ≠ утверждение заказа). Подробности — [Data Governance §«Доступ к данным»](../05-governance/03-data-governance.md).

## 8. Интеграция с Kiepper (продуктовый уровень)

### 8.1 Что читаем из Kiepper
| Сущность | Зачем | Частота обновления |
|----------|-------|---------------------|
| Справочник номенклатуры (SKU) | Базовый каталог | По изменению + ежедневная сверка |
| Рецептуры (нормы закладки) | Расчёт норм vs факт | По изменению |
| Остатки на складах | Алёрты, прогнозы | Каждые 15 минут |
| Продажи (чеки, блюда) | Прогноз спроса, отклонения | В реальном времени |
| Списания | Аномалии, отчётность | В реальном времени |
| Закупки (заказы, поступления) | План vs факт, оборачиваемость | По изменению |
| Цены (закупочные и продажи) | Маржинальность, food cost | По изменению |

### 8.2 Что пишем в Kiepper
| Сущность | Условие записи |
|----------|----------------|
| Черновик заказа поставщику | После утверждения управляющим / закупщиком |
| Комментарии к аномалиям | Опционально, по выбору пользователя |

> **Правило записи.** Orakul **никогда не пишет** мастер-данные (рецептуры, цены, справочники). Только транзакционные сущности и только после явного подтверждения.

### 8.3 Что делаем при сбое интеграции
| Сценарий | Поведение продукта |
|----------|---------------------|
| Kiepper недоступен | Работаем на последнем кэше; видим бейдж «данные на 14:30, актуальность 15 мин» |
| Стейл-данные >2ч | Скрываем рекомендации, требующие свежих остатков |
| Изменилась схема Kiepper | Запускается процедура «диагностика и блокировка» — лучше не показать, чем показать неверное |

> Подробности контракта интеграции — отдельный TRD (technical requirements document), вне периметра PRD.

## 9. Уведомления и каналы

| Тип уведомления | Когда | Где доставляется | Тихие часы |
|------------------|-------|-------------------|------------|
| Критический алёрт (стоп-лист в 24ч) | Автоматически | Push (mobile) + email | Не молчит |
| Рекомендация по заказу | Утренний слот 07:00 | Push + email | 22:00–06:00 — копится в дайджест |
| Аномалия списания | По факту | In-app + еженедельная сводка | По выбору |
| Дайджест собственника | Каждый понедельник | Email + in-app | — |
| Изменение KPI/баланса | По правилу | In-app | — |

> Принцип: **меньше — лучше**. Если за день >5 push-уведомлений на пользователя — система автоматически объединяет в дайджест и снижает приоритет (защита от banner blindness).

## 10. Нефункциональные требования (продуктовый уровень)

> Бизнес-уровень — в [BRD §7](../02-business/01-business-requirements-document.md). Здесь — продуктовая трактовка.

| Категория | Требование |
|-----------|------------|
| **Производительность** | Открытие главного экрана управляющего ≤2 сек на 4G; полный пересчёт рекомендаций по точке ≤15 мин |
| **Доступность** | Web (последние 2 версии Chrome/Safari/Firefox) + iOS/Android (последние 2 версии) + responsive для планшетов |
| **Локализация** | Русский (default), английский — задача 2027 |
| **Доступность для людей с ограниченными возможностями** | WCAG 2.1 AA как цель, AA-критичные критерии — обязательные |
| **Объяснимость** | 100% рекомендаций с полем «Обоснование», шаблон документирован |
| **Аудируемость** | Полный лог действий пользователя ≥3 года; экспорт в CSV/PDF |
| **Прайвеси и комплаенс** | Соответствие [Data Governance](../05-governance/03-data-governance.md) |
| **Reliability** | SLA доступности 99.5% в часы работы заведений (8:00–24:00 локально) |

## 12. Допущения и открытые вопросы

### 12.1 Допущения
1. У клиента есть актуальный справочник рецептур в Kiepper. Если нет — этап Data Audit обязателен (см. [Data Audit Checklist](../04-implementation/04-data-audit-checklist-kiepper.md)).
2. У клиента есть мобильные устройства для управляющих (iOS/Android последних 2 версий).
3. Часовой пояс точки определён в Kiepper (для тихих часов уведомлений).

### 12.2 Открытые вопросы (на момент v1.0)
| Вопрос | Решение к когда | Ответственный |
|--------|------------------|----------------|
| Использовать ли коммерческий LLM для генерации обоснований или собственная шаблонизация? | До конца Q1 2026 | Продакт + Data Lead |
| Поддержка офлайн-режима для управляющего на мобильном? | Решение Q3 2026 (по результатам пилотов) | Продакт |
| Нужен ли отдельный тариф «только аудит» (read-only без рекомендаций)? | После 5 платящих | CEO |
